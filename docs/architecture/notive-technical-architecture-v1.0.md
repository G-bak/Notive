# 기술 아키텍처 설계서 v1.0

# Notive Technical Architecture

---

# 1. 문서 목적

본 문서는 Notive의 1차 MVP 구현을 위한 기술 아키텍처 방향을 정의한다.

이 문서는 상세 DB 스키마, API 명세, 화면 설계, 프롬프트 설계까지 모두 다루지 않는다. 대신 각 세부 설계서가 일관된 기준으로 작성될 수 있도록 시스템 구성, 서비스 경계, 데이터 흐름, 보안 원칙, AI/검색 연동 방식, 배포 방향을 정의한다.

---

# 2. 아키텍처 목표

Notive의 초기 기술 아키텍처는 다음 목표를 가진다.

* Web SaaS 기반으로 빠르게 MVP를 구현한다.
* 조직별 데이터 분리와 권한 검사를 모든 핵심 흐름에 적용한다.
* 문서 관리, AI 생성, 검색, 업무 맥락 기능이 하나의 제품 경험으로 연결되도록 한다.
* 초기에는 단순한 구조로 시작하되, AI 서비스와 검색 기능은 향후 분리와 확장이 가능하게 설계한다.
* 문서 저장, 권한, 검색, AI 참고 자료 처리에서 보안 사고 가능성을 줄인다.
* 운영 중 오류 추적, 사용량 확인, 품질 개선이 가능하도록 로그와 지표를 남긴다.

---

# 3. 아키텍처 원칙

## 3.1 MVP 우선

초기 구조는 빠른 개발과 검증을 우선한다. 단, 이후 엔터프라이즈 확장을 막는 구조는 피한다.

---

## 3.2 권한 우선

문서, 업무 기록, 검색 결과, AI 참고 자료는 모두 권한 검사를 거쳐야 한다.

권한 검사는 화면에서만 처리하지 않고 서버 요청 처리와 데이터 조회 단계에서도 적용한다.

---

## 3.3 조직 단위 데이터 분리

모든 핵심 데이터는 조직 ID를 기준으로 분리한다.

사용자 요청은 항상 현재 조직 컨텍스트 안에서 처리되며, 다른 조직 데이터가 조회되거나 AI 참고 자료로 사용되지 않아야 한다.

---

## 3.4 문서 중심 설계

Notive의 핵심 데이터는 문서다.

업무 다이어리, To-do, 템플릿, 검색, AI 생성 결과는 모두 문서 생성과 재사용을 강화하기 위한 보조 데이터로 연결된다.

---

## 3.5 AI 결과 사용자 확인

AI 생성 결과는 자동 확정하지 않는다.

사용자가 확인, 수정, 저장한 뒤 일반 문서로 관리되도록 한다.

---

## 3.6 단계적 분리

초기에는 단일 Web 애플리케이션 중심으로 시작한다.

AI 처리, 검색 인덱싱, 문서 파싱, 장기 작업이 커지면 별도 워커 또는 서비스로 분리한다.

다만 다음 두 인프라는 B단계부터 별도 컴포넌트로 분리해 도입한다(Codex 결정).

* **Redis 단기 스토리지** — Phase D AI 미리보기 본문(미저장 24시간) 보관용. B단계는 프로비저닝과 헬스체크만 한다.
* **Background Worker / Cron** — Phase A §15 보존 정책(soft-delete 30일, AI 메타데이터 90일, AI 본문 페이로드 30일, AI 미리보기 24시간)을 강제할 정리 작업 프레임워크. B단계는 프레임워크와 dry-run 모드만 준비하고, C/D에서 실제 작업을 등록한다.

# 4. 전체 시스템 구성

## 4.1 MVP 기본 구성

MVP는 다음 구성 요소로 시작한다.

| 구성 요소 | 역할 | B단계 도입 여부 |
| --- | --- | --- |
| Web App | 사용자 화면, 앱 라우팅, 기본 서버 기능 | 도입 |
| API Layer | 인증, 권한, 문서, AI, 검색, 관리자 요청 처리 | 도입 |
| Database (PostgreSQL) | 사용자, 조직, 문서, 업무 기록, 로그 저장 | 도입 |
| Short-term Store (Redis) | AI 미리보기 본문 등 단기 보관, 세션 보조 (Phase D 사용) | B단계에 프로비저닝, D단계 사용 |
| Object Storage | 대용량 문서 첨부 또는 파일 저장 | C/이후 |
| AI Provider | 문서 생성, 요약, 문체 보정 | D단계 |
| Search Layer | 문서 검색, 자연어 검색, 검색 인덱스 관리 | F단계 |
| Background Worker / Cron | 인덱싱, 정리 작업, 로그 집계 | B단계 프레임워크, C/D 작업 등록 |
| Monitoring/Logging | 오류, 요청, 사용량 추적 | 도입 |

---

## 4.2 논리 구조

```text
User Browser
  -> Web App
    -> API Layer
      -> Auth & Permission
      -> Document Module
      -> AI Generation Module
      -> Work Context Module
      -> Search Module
      -> Admin Module
    -> Database
    -> Object Storage
    -> AI Provider
    -> Search Index
    -> Background Worker
    -> Logging/Monitoring
```

# 5. 권장 기술 구성

## 5.1 초기 MVP 권장안

| 영역 | 권장 방향 |
| --- | --- |
| Frontend | React 기반 Web App |
| Backend | Web App과 가까운 API Layer로 시작 |
| Database | PostgreSQL |
| Short-term Store | **Redis** (Codex 결정). B단계 프로비저닝, D단계 AI 미리보기 본문 보관에 사용. 세션은 Postgres 저장이 기본이며 Redis는 사용하지 않는다. |
| Background Worker / Cron | B단계 프로비저닝. C단계 문서 soft-delete 정리, D단계 AI 본문/페이로드 정리에 사용. |
| Object Storage | S3 호환 스토리지 (C/이후) |
| AI | 외부 AI API 연동 (D단계) |
| Search | 초기에는 DB 검색 + 확장 가능한 검색 인덱스 구조 (F단계) |
| Deployment | Cloud 기반 배포 |
| Monitoring | 오류 로그와 요청 로그부터 시작 |
| Auth | 이메일+비밀번호, 이메일 인증 필수, 서버 세션 (Phase A §15 / Phase B §10) |

---

## 5.2 기술 선택 기준

상세 기술 스택은 별도 구현 설계에서 확정하되, 다음 기준을 따른다.

* 팀의 개발 속도
* 유지보수 난이도
* 인증과 권한 구현의 안정성
* PostgreSQL과의 궁합
* AI API 연동 편의성
* 배포와 운영 난이도
* 향후 워커/서비스 분리 가능성
* 비용 예측 가능성

Object Storage는 대용량 파일 또는 첨부 파일 저장을 위해 사용한다.

### MVP 기준

초기 MVP에서 첨부 파일 기능이 제한적이면 Object Storage 연동은 구조만 열어둘 수 있다.

### 인덱싱 대상

# 9. 주요 요청 흐름

## 9.1 로그인 흐름

1. 사용자가 이메일과 비밀번호를 입력한다.
2. Auth Module이 계정 상태를 확인한다.
3. Membership을 조회해 조직 소속을 확인한다.
4. 세션을 생성한다.
5. 사용자 역할과 조직 컨텍스트를 앱에 제공한다.

---

## 9.2 문서 저장 흐름

1. 사용자가 문서를 작성하거나 수정한다.
2. API Layer가 세션과 조직 컨텍스트를 확인한다.
3. Permission Module이 작성/수정 권한을 확인한다.
4. Document Module이 문서를 저장한다.
5. 필요한 경우 DocumentVersion을 생성한다.
6. ActivityLog를 기록한다.
7. SearchIndex 갱신 작업을 예약한다.

---

## 9.3 AI 문서 생성 흐름

1. 사용자가 문서 유형, 템플릿, 요청 내용을 입력한다.
2. 참고 문서를 선택한다.
3. Permission Module이 참고 자료 접근 권한을 확인한다.
4. AIRequest를 생성한다.
5. AI Generation Module이 템플릿과 참고 자료를 구성한다.
6. AI Provider를 호출한다.
7. AIResult를 저장한다.
8. 사용자가 결과를 확인하고 편집기로 이동한다.
9. 사용자가 저장하면 Document로 등록된다.

---

## 9.4 업무 기록 기반 보고서 생성 흐름

1. 사용자가 기간을 선택한다.
2. DiaryEntry와 Todo 목록을 조회한다.
3. 접근 가능한 항목만 선택 가능하게 한다.
4. AIReference에 업무 맥락을 연결한다.
5. AI Generation Module이 보고서 초안을 생성한다.
6. 사용자가 편집 후 Document로 저장한다.

---

## 9.5 검색 흐름

1. 사용자가 검색어를 입력한다.
2. Search Module이 검색 대상과 필터를 해석한다.
3. 조직 ID와 권한 조건을 적용한다.
4. SearchIndex 또는 원본 데이터를 조회한다.
5. 검색 결과를 반환한다.
6. AI 요약 요청 시 권한 있는 출처만 사용한다.
7. SearchQueryLog와 SearchFeedback을 기록한다.

---

# 10. AI 아키텍처

## 10.1 AI 처리 원칙

* AI 요청은 서버에서 처리한다.
* 클라이언트는 AI Provider에 직접 접근하지 않는다.
* AI 요청 전 참고 자료 권한을 확인한다.
* AI 결과는 사용자 확인 전까지 문서로 확정하지 않는다.
* AI 요청과 결과 상태를 기록한다.

---

## 10.2 AI 요청 구성

AI 요청은 다음 요소를 조합한다.

* 사용자 요청
* 문서 유형
* 템플릿 구조
* 참고 문서
* 업무 다이어리
* To-do
* 출력 형식
* 주의 사항

---

## 10.3 AI 결과 저장

AI 결과는 AIResult로 먼저 저장된다.

사용자가 저장을 선택하면 Document로 전환되며, 이후 일반 문서와 동일한 권한, 버전, 검색 정책을 따른다.

---

## 10.4 AI 실패 처리

### 실패 유형

* AI Provider 오류
* 요청 시간 초과
* 입력 부족
* 참고 자료 권한 오류
* 사용량 제한

### 처리 원칙

* 사용자 입력값은 유지한다.
* 재시도를 제공한다.
* 실패 로그를 남긴다.
* 전체 서비스 장애로 전파하지 않는다.

---

# 11. 검색 아키텍처

## 11.1 검색 단계

검색은 다음 단계로 발전시킨다.

| 단계 | 설명 |
| --- | --- |
| 1단계 | 제목, 본문, 태그 기반 기본 검색 |
| 2단계 | 자연어 검색과 AI 요약 |
| 3단계 | 의미 기반 검색과 유사 문서 추천 |
| 4단계 | 조직 지식 그래프와 고도화된 추천 |

MVP는 1-2단계에 집중한다.

---

## 11.2 인덱싱 방식

문서 저장, 수정, 삭제 시 검색 인덱스를 갱신한다.

초기에는 동기 또는 간단한 비동기 갱신으로 시작하고, 데이터 규모가 커지면 Background Worker에서 처리한다.

---

## 11.3 검색 권한 처리

검색은 다음 기준을 반드시 적용한다.

* 조직 ID
* 문서 공유 범위
* 사용자 역할
* 팀 소속
* 개인/팀 공개 범위
* 문서 상태

---

## 11.4 AI 요약 검색

AI 요약 검색은 검색 결과 중 접근 가능한 출처만 사용한다.

출처가 부족하거나 권한상 사용할 수 있는 문서가 없으면 답변을 생성하지 않거나 제한적으로 안내한다.

---

# 12. 보안 아키텍처

## 12.1 인증

MVP는 다음 인증 구성을 사용한다(Phase A §15 잠금).

* 이메일 + 비밀번호 인증
* 이메일 인증 필수 (`users.email_verified_at`이 채워진 사용자만 `Active`로 전환)
* 서버 세션 저장 (Postgres `sessions` 테이블; 토큰은 해시로 보관)
* 비밀번호 정책: 최소 10자, 영문/숫자 혼합, 유출 비밀번호 차단 (가입 및 변경 시 검증)

향후 SSO, SAML, 2단계 인증, IP 제한을 확장할 수 있도록 계정과 인증 방식을 분리 가능한 구조로 설계한다. 다만 MVP에서는 위 구성만 구현한다.

---

## 12.2 인가

인가 판단은 다음 계층에서 모두 수행한다.

* 페이지 접근
* API 요청
* 데이터 조회
* AI 참고 자료 구성
* 검색 결과 반환

---

## 12.3 데이터 분리

모든 핵심 테이블은 조직 ID를 포함한다.

API 요청은 현재 조직 컨텍스트를 기준으로 처리하며, 다른 조직 ID가 요청에 포함되더라도 서버에서 검증한다.

---

## 12.4 로그 보안

로그에는 민감한 본문 전체를 저장하지 않는 것을 기본으로 한다.

AI 요청, 검색어, 문서 제목 등 민감할 수 있는 정보는 보존 기간과 조회 권한을 제한한다.

---

## 12.5 비밀값 관리

AI API Key, DB 접속 정보, 스토리지 키 등 비밀값은 코드에 저장하지 않는다.

환경 변수 또는 비밀값 관리 도구를 통해 관리한다.

---

# 13. 배포 아키텍처

## 13.1 MVP 배포 구조

```text
Cloud Hosting
  -> Web App / API
  -> Managed PostgreSQL
  -> Object Storage
  -> Background Worker
  -> Logging/Monitoring
  -> AI Provider
```

---

## 13.2 환경 분리

최소 다음 환경을 분리한다.

* Development
* Staging
* Production

Staging은 출시 전 테스트와 고객 데모를 위해 Production과 유사한 구조를 가진다.

---

## 13.3 배포 원칙

* Production 배포 전 Staging에서 핵심 흐름을 확인한다.
* DB 변경은 마이그레이션으로 관리한다.
* 배포 후 로그인, 문서 저장, AI 생성, 검색을 점검한다.
* 권한 관련 변경은 별도 점검 항목을 둔다.
* 롤백 기준을 사전에 정의한다.

# 15. 확장 전략

## 15.1 AI 서비스 분리

AI 요청량이 증가하거나 AI 처리 로직이 복잡해지면 AI Generation Module을 별도 서비스로 분리한다.

### 분리 기준

* AI 요청 처리 시간이 길어짐
* 큐 기반 처리 필요
* AI 모델/프롬프트 실험 증가
* 사용량과 비용 관리 고도화 필요

---

## 15.2 검색 서비스 고도화

문서 수와 검색 요구가 증가하면 별도 검색 엔진 또는 벡터 검색 구조를 도입한다.

### 전환 기준

* 기본 본문 검색 품질 부족
* 유사 문서 추천 필요
* 의미 기반 검색 필요
* 검색 속도 저하

---

## 15.3 엔터프라이즈 확장

향후 다음 요구가 확인되면 별도 아키텍처 확장을 검토한다.

* SSO/SAML
* VPC
* On-Premise
* 전용 AI 환경
* 세부 감사 로그
* 데이터 보존 정책 고도화
* 외부 협업 도구 연동
