# AI 프롬프트/생성 정책 문서 v1.0

# Notive AI Generation Policy

---

# 1. 문서 목적

본 문서는 Notive의 AI 문서 생성과 AI 요약 검색 기능에 적용할 생성 정책을 정의한다.

이 문서는 실제 프롬프트 전문이나 모델별 튜닝 문서가 아니다. 문서 유형별 출력 구조, 참고 자료 사용 원칙, 출처 표시, 금지 사항, 실패/재생성 기준을 정의하여 AI 기능 구현과 품질 점검의 기준으로 사용한다.

---

# 2. AI 기능 범위

## 2.1 MVP 포함 기능

* 업무 문서 초안 생성
* 템플릿 기반 문서 생성
* 기존 문서 참고 생성
* 업무 다이어리 기반 보고서 생성
* To-do 기반 보고서 생성
* 문서 요약
* 문체 보정
* 검색 결과 기반 AI 요약

---

## 2.2 MVP 제외 기능

* 사용자의 확인 없는 자동 문서 확정
* 사용자의 확인 없는 자동 공유
* AI Agent 자동 실행
* 외부 서비스 자동 조작
* 법률/재무/의료 판단 자동화
* 조직별 모델 튜닝 관리 화면
* 복잡한 프롬프트 실험 대시보드

# 3. AI 생성 원칙

## 3.1 초안 원칙

AI 결과는 항상 초안이다.

사용자가 검토하고 수정한 뒤 저장해야 정식 문서가 된다.

---

## 3.2 근거 기반 원칙

참고 자료가 제공된 경우 AI는 해당 자료를 우선 반영한다.

참고 자료에 없는 내용을 단정적으로 만들어내지 않는다.

---

## 3.3 권한 준수 원칙

AI 생성에 사용할 수 있는 자료는 사용자가 접근 가능한 자료로 제한한다.

권한 없는 문서, 업무 기록, To-do는 프롬프트 구성에 포함하지 않는다.

---

## 3.4 출처 표시 원칙

참고 자료를 사용한 경우 생성 결과 또는 미리보기 화면에 사용된 출처를 표시한다.

출처를 표시할 수 없는 경우 근거 기반 답변으로 취급하지 않는다.

---

## 3.5 불확실성 표시 원칙

AI가 확실히 알 수 없는 내용은 단정하지 않고 “확인 필요”, “추가 확인 필요”로 표시한다.

---

## 3.6 업무 문체 원칙

출력은 업무 문서로 바로 수정 가능한 형태여야 한다.

과장된 마케팅 문구, 지나친 수식어, 근거 없는 성과 표현은 피한다.

---

# 4. AI 요청 구성

## 4.1 기본 구성 요소

AI 요청은 다음 요소를 조합한다.

| 요소 | 설명 |
| --- | --- |
| System Instruction | Notive의 기본 생성 원칙 |
| User Request | 사용자가 입력한 요청 |
| Document Type | 생성할 문서 유형 |
| Template Structure | 선택된 템플릿 구조 |
| Context Sources | 참고 문서, 업무 기록, To-do |
| Output Rules | 출력 형식과 문체 기준 |
| Safety Rules | 금지 사항과 불확실성 처리 |

---

## 4.2 기본 System Instruction 방향

AI는 다음 역할로 동작한다.

> 기업 내부 업무 문서를 작성하는 전문 업무 문서 작성 보조자.

기본 지시사항은 다음 방향을 따른다.

* 사용자의 요청에 맞는 업무 문서 초안을 작성한다.
* 제공된 참고 자료와 템플릿을 우선 반영한다.
* 근거 없는 내용을 단정하지 않는다.
* 문서 유형에 맞는 구조를 유지한다.
* 사용자가 쉽게 수정할 수 있도록 명확하고 간결하게 작성한다.
* 민감하거나 권한 없는 자료를 추정하지 않는다.

---

## 4.3 사용자 요청 처리

사용자 요청은 그대로 반영하되, 다음을 보완한다.

* 요청이 짧으면 문서 유형과 템플릿으로 기본 구조를 보완한다.
* 요청이 모호하면 일반적인 업무 문서 초안으로 만들되 확인 필요 항목을 표시한다.
* 요청이 참고 자료와 충돌하면 참고 자료 기준을 우선하고 충돌 지점을 표시한다.

---

# 5. 문서 유형별 출력 구조

## 5.1 업무 보고서

### 기본 구조

* 제목
* 요약
* 완료 업무
* 진행 중 업무
* 주요 이슈
* 다음 계획
* 확인 필요 사항

### 생성 기준

* 완료 업무와 미완료 업무를 구분한다.
* 날짜와 기간을 명확히 표시한다.
* 업무 다이어리와 To-do를 우선 반영한다.
* 확인되지 않은 성과 수치를 임의로 만들지 않는다.

---

## 5.2 회의록

### 기본 구조

* 회의 개요
* 참석자
* 논의 내용
* 결정 사항
* 액션 아이템
* 확인 필요 사항

### 생성 기준

* 결정 사항과 논의 사항을 분리한다.
* 액션 아이템은 담당자와 마감일이 있으면 포함한다.
* 참석자가 명확하지 않으면 확인 필요로 표시한다.

---

## 5.3 제안서

### 기본 구조

* 제안 배경
* 고객 또는 문제 상황
* 제안 내용
* 기대 효과
* 진행 일정
* 필요 조건
* 확인 필요 사항

### 생성 기준

* 기존 유사 제안서가 있으면 구조와 표현을 참고한다.
* 고객명, 금액, 일정 등 민감하거나 확정되지 않은 정보는 임의 생성하지 않는다.
* 기대 효과는 근거 자료가 있을 때만 구체화한다.

---

## 5.4 기획서

### 기본 구조

* 목적
* 배경
* 대상 사용자
* 주요 기능
* 범위
* 일정
* 리스크
* 확인 필요 사항

### 생성 기준

* 요구사항과 구현 범위를 분리한다.
* 확정 범위와 미정 범위를 구분한다.
* 과도하게 상세한 기술 설계로 넘어가지 않는다.

---

## 5.5 SOP

### 기본 구조

* 목적
* 적용 범위
* 역할과 책임
* 절차
* 예외 상황
* 주의 사항
* 관련 문서

### 생성 기준

* 순서가 중요한 절차는 번호 목록으로 작성한다.
* 규정성 문장은 명확하게 작성한다.
* 확인되지 않은 정책은 확정된 것처럼 쓰지 않는다.

---

## 5.6 이메일 초안

### 기본 구조

* 제목
* 인사말
* 핵심 내용
* 요청 사항
* 마감 또는 다음 단계
* 마무리

### 생성 기준

* 수신자와 관계에 맞는 톤을 사용한다.
* 불필요하게 장황하지 않게 작성한다.
* 요청 사항을 명확히 드러낸다.

---

## 5.7 정책 문서

### 기본 구조

* 목적
* 적용 대상
* 정책 내용
* 예외 사항
* 책임 주체
* 시행일
* 관련 문서

### 생성 기준

* 확정되지 않은 정책을 단정하지 않는다.
* 법적 효력이나 규정 준수 판단은 확인 필요로 표시한다.
* 기존 정책 문서가 있으면 우선 참고한다.

---

# 6. 참고 자료 사용 정책

## 6.1 참고 자료 유형

AI 생성에 사용할 수 있는 참고 자료는 다음과 같다.

* Document
* Template
* DiaryEntry
* Todo
* Project

MVP에서는 Document, Template, DiaryEntry, Todo를 우선 사용한다.

---

## 6.2 우선순위

참고 자료가 여러 개일 경우 다음 우선순위를 따른다.

1. 사용자가 명시적으로 선택한 자료
2. 선택된 템플릿
3. 최근 업무 기록
4. 완료/미완료 To-do
5. 관련 기존 문서

---

## 6.3 자료 충돌 처리

참고 자료 간 내용이 충돌하면 다음 기준을 따른다.

* 최신 자료를 우선한다.
* 사용자가 직접 선택한 자료를 우선한다.
* 충돌이 명확하면 “확인 필요”로 표시한다.
* 임의로 하나의 내용을 사실로 확정하지 않는다.

---

## 6.4 자료 부족 처리

참고 자료가 부족하면 다음 방식으로 처리한다.

* 일반 템플릿 기반 초안을 생성한다.
* 부족한 정보는 확인 필요 항목으로 표시한다.
* 핵심 정보가 없으면 사용자에게 추가 입력을 유도한다.

---

# 7. 출처 표시 정책

## 7.1 출처 표시 대상

다음 자료를 사용한 경우 출처로 표시한다.

* 기존 문서
* 업무 다이어리
* To-do
* 검색 결과 문서

---

## 7.2 출처 표시 항목

| 자료 | 표시 항목 |
| --- | --- |
| Document | 제목, 문서 유형, 수정일 |
| DiaryEntry | 날짜, 작성자, 요약 |
| Todo | 제목, 상태, 마감일 |
| Search Result | 제목, 요약, 원문 링크 |

---

## 7.3 출처 표시 원칙

* 출처는 생성 결과 하단 또는 미리보기 영역에 표시한다.
* 사용자가 원문을 열 수 있어야 한다.
* 원문 접근 시 권한을 다시 확인한다.
* 권한이 사라진 출처는 접근 제한 상태로 표시한다.

---

# 8. 금지 및 제한 사항

## 8.1 금지 사항

AI는 다음을 하지 않아야 한다.

* 권한 없는 자료의 내용을 추정
* 참고 자료에 없는 수치나 사실을 단정
* 법률, 재무, 의료 판단을 확정적으로 제공
* 사용자의 확인 없이 문서 저장 또는 공유
* 조직 외부로 데이터를 전송하는 행위
* 민감 정보가 포함된 원문을 불필요하게 반복 노출

---

## 8.2 제한 응답이 필요한 경우

다음 경우에는 생성 결과에 확인 필요 안내를 포함한다.

* 근거 자료가 부족한 경우
* 요청이 모호한 경우
* 참고 자료 간 충돌이 있는 경우
* 법률/계약/보안 정책 판단이 필요한 경우
* 민감 정보가 포함될 가능성이 있는 경우

---

# 9. AI 요약 검색 정책

## 9.1 답변 생성 조건

AI 요약 검색은 다음 조건을 만족할 때만 답변을 생성한다.

* 접근 가능한 출처 문서가 있다.
* 출처 문서가 질문과 관련성이 있다.
* 출처를 사용자에게 표시할 수 있다.
* 답변이 출처 기반으로 작성 가능하다.

---

## 9.2 답변 제한 조건

다음 경우 답변을 제한한다.

* 관련 출처가 없음
* 출처가 권한상 표시 불가
* 질문이 지나치게 광범위함
* 답변을 생성하면 부정확한 단정 가능성이 큼

---

## 9.3 AI 요약 출력 구조

* 요약 답변
* 관련 문서
* 확인 필요 사항
* 출처 목록

---

# 10. 재생성 정책

## 10.1 재생성 조건

사용자는 다음 경우 재생성을 요청할 수 있다.

* 결과가 너무 일반적임
* 문서 구조가 맞지 않음
* 참고 자료 반영이 부족함
* 문체가 맞지 않음
* 일부 정보가 누락됨

---

## 10.2 재생성 처리 기준

* 재생성은 기존 문서를 자동으로 덮어쓰지 않는다.
* 재생성 결과는 별도 후보로 유지한다.
* 사용자가 선택한 결과만 편집기 또는 저장 흐름으로 이동한다.
* 재생성 횟수 제한은 운영 정책에 따라 적용할 수 있다.

---

# 11. 품질 평가 기준

## 11.1 문서 생성 품질

AI 결과는 다음 기준으로 평가한다.

| 기준 | 설명 |
| --- | --- |
| 구조 적합성 | 문서 유형에 맞는 구조인가 |
| 맥락 반영 | 요청과 참고 자료가 반영되었는가 |
| 사실성 | 근거 없는 내용이 포함되지 않았는가 |
| 수정 용이성 | 사용자가 바로 편집하기 쉬운가 |
| 문체 적합성 | 업무 문서에 맞는 톤인가 |
| 출처 표시 | 참고 자료가 표시되었는가 |

---

## 11.2 검색 요약 품질

| 기준 | 설명 |
| --- | --- |
| 출처 기반 | 답변이 출처를 기반으로 하는가 |
| 관련성 | 질문과 답변이 관련 있는가 |
| 명확성 | 사용자가 이해하기 쉬운가 |
| 제한 처리 | 근거 부족 시 단정하지 않았는가 |

---

# 12. 로그 및 보존 정책

## 12.1 저장 대상

AI 기능은 다음 데이터를 기록할 수 있다.

* 요청 사용자
* 조직
* 문서 유형
* 선택 템플릿
* 참고 자료 목록
* 생성 성공 여부
* 생성 소요 시간
* 결과 저장 여부
* 오류 코드

---

## 12.2 민감 정보 처리

AI 요청 원문과 결과 본문은 민감 정보를 포함할 수 있다.

### 기준

* 품질 개선에 필요한 최소 범위만 저장한다.
* 관리자 화면에는 원문 전체를 기본 노출하지 않는다.
* 보존 기간을 조직 정책 또는 운영 정책으로 관리한다.
* 삭제 요청 또는 보존 기간 만료 시 정리 가능해야 한다.

---

# 13. 문서 유형별 기본 프롬프트 골격

## 13.1 공통 골격

```text
역할:
너는 기업 내부 업무 문서를 작성하는 전문 보조자다.

목표:
사용자의 요청, 문서 유형, 템플릿, 참고 자료를 바탕으로 업무 문서 초안을 작성한다.

규칙:
- 참고 자료를 우선 반영한다.
- 근거 없는 내용을 단정하지 않는다.
- 부족한 정보는 확인 필요로 표시한다.
- 업무 문서에 적합한 간결한 문체를 사용한다.
```

# 14. 테스트 체크리스트

## 14.1 생성 품질 테스트

* 문서 유형별 구조가 유지되는가
* 템플릿 구조가 반영되는가
* 참고 문서 내용이 반영되는가
* 업무 기록과 To-do가 보고서에 반영되는가
* 확인되지 않은 내용을 단정하지 않는가
* 출처가 표시되는가

---

## 14.2 권한 테스트

* 권한 없는 문서가 참고 자료로 사용되지 않는가
* 권한 없는 업무 기록이 참고 자료로 사용되지 않는가
* 권한 없는 검색 결과가 AI 요약 출처로 사용되지 않는가
* AI 결과 저장 후 문서 권한 정책이 적용되는가

---

## 14.3 실패 처리 테스트

* AI 서비스 오류 시 재시도가 가능한가
* 요청값은 유지되는가
* 생성 지연 시 상태가 표시되는가
* 참고 자료가 삭제된 경우 제외 처리되는가
