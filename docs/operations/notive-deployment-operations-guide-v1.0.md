# 배포/운영 가이드 v1.0

# Notive Deployment & Operations Guide

---

# 1. 문서 목적

본 문서는 Notive MVP의 배포와 운영 기준을 정의한다.

이 문서는 특정 클라우드 벤더에 종속된 상세 매뉴얼이 아니다. 개발, 스테이징, 운영 환경 구성 원칙과 배포 전후 점검, 롤백, 모니터링, 장애 대응, 운영 업무의 기준을 정리한다.

---

# 2. 운영 목표

Notive MVP 운영의 목표는 다음과 같다.

* 초기 고객 또는 내부 테스트 그룹이 안정적으로 서비스를 사용할 수 있게 한다.
* 로그인, 문서 저장, AI 생성, 검색의 장애를 빠르게 발견한다.
* 권한 오류와 데이터 노출 가능성을 최우선으로 관리한다.
* 배포 실패 시 빠르게 이전 안정 상태로 되돌릴 수 있다.
* 사용자 피드백과 오류 로그를 다음 개선 작업으로 연결한다.

---

# 3. 환경 구성

## 3.1 환경 구분

| 환경 | 목적 |
| --- | --- |
| Development | 개발자 로컬 또는 개발 서버 |
| Staging | 출시 전 검증과 데모 |
| Production | 실제 사용자 서비스 |

---

## 3.2 환경별 원칙

### Development

* 빠른 개발과 테스트를 우선한다.
* 샘플 데이터를 사용할 수 있다.
* 실제 고객 데이터는 사용하지 않는다.

### Staging

* Production과 최대한 유사한 설정을 사용한다.
* 배포 전 핵심 흐름을 검증한다.
* 테스트 계정과 샘플 조직을 유지한다.

### Production

* 실제 사용자 데이터가 저장된다.
* 비밀값과 접근 권한을 엄격히 관리한다.
* 배포 전후 점검과 롤백 기준을 따른다.

---

# 4. 기본 인프라 구성

## 4.1 MVP 구성 요소

| 구성 요소 | 설명 |
| --- | --- |
| Web App/API | 사용자 화면과 API 요청 처리 |
| PostgreSQL | 핵심 데이터 저장 |
| Object Storage | 첨부 파일 또는 원본 파일 저장 |
| Background Worker | 인덱싱, 집계, 장기 작업 처리 |
| AI Provider | AI 문서 생성과 요약 처리 |
| Logging/Monitoring | 오류와 사용량 추적 |

---

## 4.2 필수 비밀값

| 항목 | 설명 |
| --- | --- |
| DATABASE_URL | DB 접속 정보 |
| SESSION_SECRET | 세션 또는 토큰 서명 키 |
| AI_API_KEY | AI Provider API Key |
| STORAGE_ACCESS_KEY | Object Storage 접근 키 |
| STORAGE_SECRET_KEY | Object Storage 비밀 키 |
| APP_BASE_URL | 서비스 기본 URL |

비밀값은 코드 저장소에 커밋하지 않는다.

# 5. 배포 전 준비

## 5.1 코드 준비

* 배포 대상 브랜치 확인
* 주요 변경 사항 확인
* 마이그레이션 포함 여부 확인
* 환경 변수 변경 여부 확인
* 권한 관련 변경 여부 확인
* AI/검색 관련 변경 여부 확인

---

## 5.2 데이터베이스 준비

* 마이그레이션 실행 순서 확인
* 롤백 가능 여부 확인
* 위험한 데이터 변경 여부 확인
* 운영 데이터 백업 여부 확인
* 인덱스 추가 시 영향 확인

---

## 5.3 Staging 점검

Production 배포 전 Staging에서 다음을 확인한다.

* 로그인
* 조직 선택
* 문서 작성/저장
* AI 문서 생성
* 검색
* 관리자 화면
* 권한 차단
* 주요 오류 로그

---

# 6. 배포 절차

## 6.1 기본 배포 흐름

1. 배포 대상 버전을 확정한다.
2. Staging에 배포한다.
3. Staging 핵심 흐름을 점검한다.
4. Production 데이터 백업 상태를 확인한다.
5. Production에 배포한다.
6. 마이그레이션을 실행한다.
7. 배포 후 확인 항목을 점검한다.
8. 오류 로그와 주요 지표를 모니터링한다.

---

## 6.2 배포 중 주의 사항

* DB 마이그레이션은 되돌리기 어려울 수 있으므로 사전 검토한다.
* 권한 정책 변경은 별도 테스트를 수행한다.
* AI 요청 처리 구조 변경은 실패율을 확인한다.
* 검색 인덱스 구조 변경은 검색 결과 누락 여부를 확인한다.

# 8. 롤백 기준

## 8.1 즉시 롤백 검토 상황

* 로그인 장애
* 문서 저장 장애
* 권한 없는 데이터 노출
* 다른 조직 데이터 접근 가능
* 관리자 권한 관리 불가
* AI 요청이 전체 서비스 장애를 유발
* 주요 화면 접근 불가

---

## 8.2 롤백 전 확인

* 코드 롤백으로 해결 가능한 문제인가
* DB 마이그레이션이 롤백 가능한가
* 데이터 손상 가능성이 있는가
* 일시적인 외부 서비스 장애인가
* 기능 플래그 또는 설정 변경으로 우회 가능한가

---

## 8.3 롤백 후 확인

* 로그인 가능 여부
* 문서 저장 가능 여부
* 권한 차단 정상 여부
* 오류 로그 감소 여부
* 사용자 영향 범위 기록

---

| P2 | 일부 기능 오류, 특정 문서 유형 AI 품질 문제 |
| P3 | 사용성 개선, 문구 수정, 비핵심 지표 지연 |

# 10. 장애 대응

## 10.1 장애 등급

| 등급 | 설명 | 예시 |
| --- | --- | --- |
| Sev1 | 서비스 핵심 기능 중단 또는 데이터 노출 | 로그인 불가, 문서 저장 불가, 권한 없는 문서 노출 |
| Sev2 | 주요 기능 일부 중단 | AI 생성 불가, 검색 불가 |
| Sev3 | 제한적 기능 오류 | 특정 필터 오류, 일부 관리자 지표 오류 |
| Sev4 | 사용성 또는 문구 문제 | 안내 문구 오류, 레이아웃 개선 |

---

## 10.2 장애 대응 흐름

1. 장애를 감지한다.
2. 영향 범위를 파악한다.
3. 등급을 지정한다.
4. 임시 대응 또는 롤백 여부를 결정한다.
5. 사용자 공지 필요 여부를 판단한다.
6. 원인을 분석한다.
7. 재발 방지 작업을 등록한다.

---

## 10.3 데이터 노출 의심 대응

권한 없는 문서 또는 조직 데이터 노출이 의심되면 다음 순서로 대응한다.

1. 관련 기능을 즉시 중단하거나 접근을 제한한다.
2. 영향 조직과 사용자 범위를 확인한다.
3. 로그를 보존한다.
4. 원인을 분석한다.
5. 수정 후 권한 테스트를 재실행한다.
6. 필요 시 고객에게 공지한다.

# 11. 백업 및 복구

## 11.1 백업 대상

* PostgreSQL 데이터
* Object Storage 파일
* 환경 설정
* 주요 운영 로그

---

## 11.2 백업 기준

MVP에서는 최소 일 단위 백업을 권장한다.

문서 저장 안정성이 중요하므로, 운영 환경에서는 백업 상태를 정기적으로 확인한다.

---

## 11.3 복구 점검

백업은 존재만으로 충분하지 않다.

정기적으로 복구 가능 여부를 점검한다.

### 점검 항목

* DB 복구 가능 여부
* 특정 문서 복구 가능 여부
* Object Storage 파일 접근 가능 여부
* 복구 후 권한 유지 여부

# 13. 보안 운영

## 13.1 접근 권한 관리

* Production 접근 권한은 최소 인원에게만 부여한다.
* DB 직접 접근은 제한한다.
* 비밀값 접근 권한을 관리한다.
* 퇴사자 또는 역할 변경자의 접근 권한을 회수한다.

---

## 13.2 비밀값 관리

* 비밀값은 코드 저장소에 저장하지 않는다.
* 키가 노출되면 즉시 교체한다.
* 운영과 개발 환경의 키를 분리한다.
* AI API Key 사용량을 주기적으로 확인한다.

---

## 13.3 로그 관리

* 문서 본문과 AI 요청 원문을 로그에 과도하게 남기지 않는다.
* 검색어 원문은 민감 정보가 될 수 있다.
* 로그 보존 기간을 관리한다.
* 관리자 로그 조회 권한을 제한한다.

---
