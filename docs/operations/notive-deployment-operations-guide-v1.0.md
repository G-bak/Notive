# 배포/운영 가이드 v1.0

# Notive Deployment & Operations Guide

---

# 1. 문서 목적

본 문서는 Notive MVP의 배포와 운영 기준을 정의한다.

이 문서는 특정 클라우드 벤더에 종속된 상세 매뉴얼이 아니다. 개발, 스테이징, 운영 환경 구성 원칙과 배포 전후 점검, 롤백, 모니터링, 장애 대응, 운영 업무의 기준을 정리한다.

---

# 2. 운영 목표

Notive MVP 운영의 목표는 다음과 같다.

* 초기 고객 또는 내부 테스트 그룹이 안정적으로 서비스를 사용할 수 있게 한다.
* 로그인, 문서 저장, AI 생성, 검색의 장애를 빠르게 발견한다.
* 권한 오류와 데이터 노출 가능성을 최우선으로 관리한다.
* 배포 실패 시 빠르게 이전 안정 상태로 되돌릴 수 있다.
* 사용자 피드백과 오류 로그를 다음 개선 작업으로 연결한다.

---

# 3. 환경 구성

## 3.1 환경 구분

| 환경 | 목적 |
| --- | --- |
| Development | 개발자 로컬 또는 개발 서버 |
| Staging | 출시 전 검증과 데모 |
| Production | 실제 사용자 서비스 |

---

## 3.2 환경별 원칙

### Development

* 빠른 개발과 테스트를 우선한다.
* 샘플 데이터를 사용할 수 있다.
* 실제 고객 데이터는 사용하지 않는다.

### Staging

* Production과 최대한 유사한 설정을 사용한다.
* 배포 전 핵심 흐름을 검증한다.
* 테스트 계정과 샘플 조직을 유지한다.

### Production

* 실제 사용자 데이터가 저장된다.
* 비밀값과 접근 권한을 엄격히 관리한다.
* 배포 전후 점검과 롤백 기준을 따른다.

# 4. 기본 인프라 구성

## 4.1 MVP 구성 요소

스택 잠금: 아키텍처 §5.1 참조. 본 절은 그 결정을 운영 관점에서 정리한다.

| 구성 요소 | 설명 | 도입 단계 |
| --- | --- | --- |
| Web/API App | Next.js App Router (TypeScript, Node.js LTS) 단일 컨테이너. App Router의 서버 컴포넌트 + Route Handler가 API 역할을 겸한다. | B단계 |
| Worker Process | Node.js LTS 프로세스. Web/API와 같은 이미지의 다른 entrypoint로 실행한다. cron 스케줄과 dry-run 모드 지원. | B단계 |
| PostgreSQL | Managed Postgres. DB 설계 문서가 스키마 단일 소스이고 Prisma 마이그레이션은 여기서 파생된다. 사용자/조직/문서/세션을 모두 보관. | B단계 |
| Redis-compatible | AI 미리보기 본문 단기 보관 (24시간 idle TTL). 세션은 Postgres에 저장하므로 Redis는 사용하지 않는다. 구체 매니지드 서비스(예: ElastiCache, Upstash, self-hosted)는 B단계 인프라 구현 시 Codex 검증 대상. | B단계 프로비저닝, D단계 사용 |
| Object Storage | 첨부 파일 또는 원본 파일 저장 | C/이후 |
| AI Provider | AI 문서 생성과 요약 처리 | D단계 |
| Mail Provider | 가입 인증 메일, 초대 메일 | B단계 |
| Logging/Monitoring | 오류와 사용량 추적. 구체 도구(예: Sentry, OpenTelemetry collector)는 B단계 인프라 결정. | B단계 |

### 컨테이너화 가정

* Web/API와 Worker는 같은 코드 베이스에서 빌드된 단일 이미지의 다른 entrypoint를 사용한다(예: `pnpm start:web`, `pnpm start:worker`).
* 컨테이너 런타임은 Node.js LTS. lockfile은 `pnpm-lock.yaml`을 사용한다.
* 빌드 산출물은 `pnpm build`로 만들고 마이그레이션은 컨테이너 부팅 전에 별도 단계 또는 init 컨테이너에서 `prisma migrate deploy`로 적용한다.

# 6. 배포 절차

## 6.1 기본 배포 흐름

1. 배포 대상 버전을 확정한다.
2. Staging에 배포한다.
3. Staging 핵심 흐름을 점검한다.
4. Production 데이터 백업 상태를 확인한다.
5. Production에 배포한다.
6. 마이그레이션을 실행한다.
7. 배포 후 확인 항목을 점검한다.
8. 오류 로그와 주요 지표를 모니터링한다.

---

## 6.2 배포 중 주의 사항

* DB 마이그레이션은 되돌리기 어려울 수 있으므로 사전 검토한다.
* 권한 정책 변경은 별도 테스트를 수행한다.
* AI 요청 처리 구조 변경은 실패율을 확인한다.
* 검색 인덱스 구조 변경은 검색 결과 누락 여부를 확인한다.

# 8. 롤백 기준

## 8.1 즉시 롤백 검토 상황

* 로그인 장애
* 문서 저장 장애
* 권한 없는 데이터 노출
* 다른 조직 데이터 접근 가능
* 관리자 권한 관리 불가
* AI 요청이 전체 서비스 장애를 유발
* 주요 화면 접근 불가

---

## 8.2 롤백 전 확인

* 코드 롤백으로 해결 가능한 문제인가
* DB 마이그레이션이 롤백 가능한가
* 데이터 손상 가능성이 있는가
* 일시적인 외부 서비스 장애인가
* 기능 플래그 또는 설정 변경으로 우회 가능한가

---

## 8.3 롤백 후 확인

* 로그인 가능 여부
* 문서 저장 가능 여부
* 권한 차단 정상 여부
* 오류 로그 감소 여부
* 사용자 영향 범위 기록

# 10. 장애 대응

## 10.1 장애 등급

| 등급 | 설명 | 예시 |
| --- | --- | --- |
| Sev1 | 서비스 핵심 기능 중단 또는 데이터 노출 | 로그인 불가, 문서 저장 불가, 권한 없는 문서 노출 |
| Sev2 | 주요 기능 일부 중단 | AI 생성 불가, 검색 불가 |
| Sev3 | 제한적 기능 오류 | 특정 필터 오류, 일부 관리자 지표 오류 |
| Sev4 | 사용성 또는 문구 문제 | 안내 문구 오류, 레이아웃 개선 |

---

## 10.2 장애 대응 흐름

1. 장애를 감지한다.
2. 영향 범위를 파악한다.
3. 등급을 지정한다.
4. 임시 대응 또는 롤백 여부를 결정한다.
5. 사용자 공지 필요 여부를 판단한다.
6. 원인을 분석한다.
7. 재발 방지 작업을 등록한다.

---

## 10.3 데이터 노출 의심 대응

권한 없는 문서 또는 조직 데이터 노출이 의심되면 다음 순서로 대응한다.

1. 관련 기능을 즉시 중단하거나 접근을 제한한다.
2. 영향 조직과 사용자 범위를 확인한다.
3. 로그를 보존한다.
4. 원인을 분석한다.
5. 수정 후 권한 테스트를 재실행한다.
6. 필요 시 고객에게 공지한다.

# 11. 백업 및 복구

## 11.1 백업 대상

* 환경 설정

## 11.2 백업 기준

MVP에서는 최소 일 단위 백업을 권장한다.

문서 저장 안정성이 중요하므로, 운영 환경에서는 백업 상태를 정기적으로 확인한다.

---

## 11.3 복구 점검

백업은 존재만으로 충분하지 않다.

정기적으로 복구 가능 여부를 점검한다.

### 점검 항목

* DB 복구 가능 여부
* 특정 문서 복구 가능 여부
* Object Storage 파일 접근 가능 여부
* 복구 후 권한 유지 여부

# 13. 보안 운영

## 13.1 접근 권한 관리

* Production 접근 권한은 최소 인원에게만 부여한다.
* DB 직접 접근은 제한한다.
* 비밀값 접근 권한을 관리한다.
* 퇴사자 또는 역할 변경자의 접근 권한을 회수한다.

---

## 13.2 비밀값 관리

* 비밀값은 코드 저장소에 저장하지 않는다.
* 키가 노출되면 즉시 교체한다.
* 운영과 개발 환경의 키를 분리한다.
* AI API Key 사용량을 주기적으로 확인한다.

---

## 13.3 로그 관리

* 문서 본문과 AI 요청 원문을 로그에 과도하게 남기지 않는다.
* 검색어 원문은 민감 정보가 될 수 있다.
* 로그 보존 기간을 관리한다.
* 관리자 로그 조회 권한을 제한한다.

# 13.4 정리 작업 (Cleanup Workers)

Phase A §15가 정의한 보존 정책은 백그라운드 정리 작업으로 강제한다. 작업 프레임워크는 B단계에서 프로비저닝되며, 각 작업은 도입 단계에서 등록된다.

| 작업 | 주기 | 기준 | 도입 단계 |
| --- | --- | --- | --- |
| AI 미리보기 본문 정리 | 5분 | Redis 단기 스토리지에서 24시간 idle 또는 명시적 폐기 | D단계 |
| AI 본문 페이로드 정리 | 매일 | `ai_request_payloads.retain_until < now` 인 row 물리 삭제 | D단계 |
| AI 메타데이터 정리 | 매일 | `ai_requests` / `ai_results` / `ai_usage_logs` / `ai_references` 90일 초과 row 삭제 | D단계 |
| 문서 soft-delete 정리 | 매일 | `documents.deleted_at + 30일 < now` 인 row 물리 삭제 | C단계 |
| 세션 만료 정리 | 매시간 | `sessions.expires_at < now` 또는 `revoked_at` 채워진 row 정리 | B단계 |

### 운영 원칙

* B단계는 작업 프레임워크와 dry-run 모드만 준비한다. 비즈니스 작업은 C/D에서 등록한다.
* `WORKER_DESTRUCTIVE_OPS` 환경 변수가 명시적으로 `true`인 경우에만 실제 삭제가 일어난다. 기본은 `false`(dry-run)이다.
* 모든 정리 작업은 idempotent해야 하며, 같은 row를 두 번 처리해도 동일 결과여야 한다.
* 작업 실행 시 시작/완료 시각, 처리 row 수, 오류를 운영 로그에 남긴다.
* P0 알림: 정리 작업 실패가 24시간 누적되면 알림. AI 페이로드 30일 정리 누락은 보안 이슈로 처리한다.
