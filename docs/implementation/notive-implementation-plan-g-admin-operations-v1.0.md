# G. 관리자 및 운영 기능 세부 구현 계획서 v1.0

# Notive 관리자 및 운영 기능

---

# 1. 문서 목적

본 문서는 Notive 전체 구현 계획 중 G단계인 관리자 및 운영 기능 구현의 세부 계획을 정의한다.

G단계의 목적은 조직 관리자가 사용자, 팀, 권한, 템플릿, 사용 현황, 주요 활동 로그를 관리할 수 있도록 하는 것이다. 또한 서비스 운영자가 초기 고객 운영과 문제 대응에 필요한 기본 정보를 확인할 수 있는 기반을 마련한다.

---

# 2. G단계 목표

G단계가 완료되면 다음이 가능해야 한다.

* Admin이 조직 사용자를 초대, 조회, 비활성화할 수 있다.
* Admin이 팀/부서를 생성하고 사용자를 배정할 수 있다.
* Admin이 사용자 역할을 변경할 수 있다.
* Admin이 문서 템플릿을 등록, 수정, 비활성화할 수 있다.
* Manager는 허용된 범위에서 팀 사용자를 확인할 수 있다.
* 주요 활동 로그를 조회할 수 있다.
* AI 요청과 검색 사용 현황을 기본 수준에서 확인할 수 있다.
* 조직 기본 설정과 보안 관련 옵션을 관리할 수 있다.

# 5. 주요 사용자 흐름

## 5.1 사용자 초대

### 기본 흐름

1. Admin이 관리자 화면에서 사용자 초대를 선택한다.
2. 초대할 이메일을 입력한다.
3. 역할과 팀을 지정한다.
4. 초대를 발송한다.
5. 초대 대기 상태가 사용자 목록에 표시된다.
6. 초대 대상자가 수락하면 Active 사용자로 전환된다.

### 예외 흐름

* 이미 조직에 속한 이메일이면 중복 안내를 표시한다.
* 만료된 초대는 재발송할 수 있다.
* 잘못 발송한 초대는 취소할 수 있다.
* 초대 권한이 없는 사용자는 초대 화면에 접근할 수 없다.

---

## 5.2 사용자 비활성화

### 기본 흐름

1. Admin이 사용자 목록에서 사용자를 선택한다.
2. 비활성화를 선택한다.
3. 비활성화 영향 안내를 확인한다.
4. 사용자가 Disabled 상태로 변경된다.
5. 해당 사용자는 더 이상 로그인할 수 없다.

### 설계 기준

* 비활성화된 사용자의 작성 문서는 유지한다.
* 문서 소유권 이전 기능은 후속으로 제공하거나 Admin 수동 처리 기준을 둔다.
* 자기 자신을 비활성화할 수 없도록 한다.
* 마지막 Admin 비활성화는 차단한다.

---

## 5.3 역할 변경

### 기본 흐름

1. Admin이 사용자 상세 또는 목록에서 역할 변경을 선택한다.
2. 새 역할을 선택한다.
3. 변경 영향을 확인한다.
4. 역할이 변경된다.
5. 변경 사항이 활동 로그에 기록된다.

### 예외 흐름

* Admin이 아닌 사용자는 역할을 변경할 수 없다.
* 마지막 Admin을 다른 역할로 낮출 수 없다.
* 자기 자신의 Admin 권한 제거는 제한할 수 있다.

---

## 5.4 팀/부서 관리

### 기본 흐름

1. Admin이 팀/부서 관리 화면에 진입한다.
2. 새 팀을 생성한다.
3. 팀명을 입력한다.
4. 팀 관리자를 지정한다.
5. 사용자를 팀에 배정한다.
6. 팀 정보가 사용자, 문서, 업무 기록 권한에 반영된다.

### 예외 흐름

* 팀명이 비어 있으면 저장할 수 없다.
* 사용자가 여러 팀에 속할 수 있는지 여부는 A단계 결정 사항을 따른다.
* 팀 삭제 시 소속 사용자와 문서 처리 정책을 안내한다.

---

## 5.5 템플릿 관리

### 기본 흐름

1. Admin이 템플릿 관리 화면에 진입한다.
2. 새 템플릿을 등록한다.
3. 문서 유형, 적용 팀, 템플릿 본문 구조를 입력한다.
4. 사용 여부를 설정한다.
5. AI 문서 생성과 일반 문서 작성에서 템플릿을 사용할 수 있다.

### 예외 흐름

* 비활성화된 템플릿은 신규 생성에 노출하지 않는다.
* 기존 문서에는 템플릿 변경 사항을 자동 반영하지 않는다.
* 팀 전용 템플릿은 해당 팀 사용자에게만 노출한다.

---

## 5.6 활동 로그 조회

### 기본 흐름

1. Admin이 활동 로그 화면에 진입한다.
2. 기간, 사용자, 활동 유형으로 필터링한다.
3. 주요 활동 내역을 확인한다.
4. 필요한 경우 특정 활동의 상세를 확인한다.

### 주요 로그 유형

* 로그인
* 초대 발송
* 초대 수락
* 역할 변경
* 팀 생성/수정
* 문서 생성/수정/삭제
* 문서 공유 변경
* AI 생성 요청
* 검색 요청

---

## 5.7 사용 현황 확인

### 기본 흐름

1. Admin이 관리자 대시보드에 진입한다.
2. 사용자 수, 문서 수, AI 요청 수, 검색 수를 확인한다.
3. 최근 주요 활동을 확인한다.
4. 이상 사용량이나 오류가 있는지 확인한다.

### MVP 기준

초기에는 운영 판단을 위한 기본 숫자와 최근 활동 중심으로 제공한다. 복잡한 분석 대시보드는 후속으로 미룬다.

| 생성 문서 수 | 문서 활용도 확인 |
| AI 요청 수 | AI 기능 사용량 확인 |
| AI 실패 수 | 품질 및 안정성 확인 |
| 검색 요청 수 | 지식 검색 활용도 확인 |
| 검색 피드백 | 검색 품질 확인 |
| 초대 대기 수 | 온보딩 진행 확인 |

## 10.2 운영자 관점 지표

서비스 운영자는 고객 지원과 안정성을 위해 다음 지표를 확인할 수 있어야 한다.

* 조직별 사용자 수
* 조직별 문서 수
* 조직별 AI 요청 수
* AI 실패율
* 검색 실패 또는 결과 없음 비율
* 최근 오류 발생 현황

MVP에서는 운영자 전용 백오피스가 없더라도 로그와 집계 데이터 구조를 준비한다.

# 12. 오류 및 예외 처리

## 12.1 사용자 관리 오류

* 중복 초대
* 존재하지 않는 사용자
* 마지막 Admin 비활성화 시도
* 자기 자신 권한 제거 시도
* 비활성화된 사용자 역할 변경

### UX 기준

* 위험한 작업은 확인 단계를 둔다.
* 실패 이유를 관리자에게 명확히 표시한다.
* 보안상 내부 오류 상세는 노출하지 않는다.

---

## 12.2 팀 관리 오류

* 중복 팀명
* 팀 삭제 시 소속 사용자 존재
* 팀 삭제 시 소유 문서 존재
* 존재하지 않는 팀 관리자 지정

---

## 12.3 템플릿 관리 오류

* 템플릿명 누락
* 문서 유형 누락
* 비어 있는 템플릿 본문
* 사용 중인 템플릿 삭제 시도

### 대응 기준

* 삭제보다 비활성화를 우선한다.
* 기존 문서에는 템플릿 변경을 자동 반영하지 않는다.
* AI 생성에서 사용 중인 템플릿 상태를 확인한다.

* 문서 생성 수가 대시보드에 반영되는지 확인

# 14. 완료 기준

G단계는 다음 기준을 만족하면 완료로 본다.

* Admin이 사용자 목록을 확인할 수 있다.
* Admin이 사용자를 초대하고 초대를 취소할 수 있다.
* Admin이 사용자를 비활성화할 수 있다.
* Admin이 사용자 역할을 변경할 수 있다.
* 마지막 Admin 제거가 차단된다.
* Admin이 팀/부서를 생성하고 사용자를 배정할 수 있다.
* Admin이 템플릿을 등록, 수정, 비활성화할 수 있다.
* 주요 활동 로그를 조회할 수 있다.
* 관리자 대시보드에서 기본 사용 현황을 확인할 수 있다.
* Manager와 Admin의 접근 범위가 구분된다.
* 조직 설정과 기본 보안 설정의 일부를 관리할 수 있다.
