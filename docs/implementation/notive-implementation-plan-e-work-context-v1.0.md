# E. 업무 맥락 기능 세부 구현 계획서 v1.0

# Notive 업무 맥락 기능

---

# 1. 문서 목적

본 문서는 Notive 전체 구현 계획 중 E단계인 업무 맥락 기능 구현의 세부 계획을 정의한다.

E단계의 목적은 사용자의 업무 다이어리, To-do, 프로젝트/태그 정보를 문서 생성에 활용 가능한 업무 맥락 데이터로 구축하는 것이다. 이 단계는 Notive가 단순 문서 생성기를 넘어 실제 업무 흐름을 이해하는 플랫폼으로 확장되는 핵심 단계다.

---

# 2. E단계 목표

E단계가 완료되면 다음이 가능해야 한다.

* 사용자가 날짜별 업무 기록을 작성할 수 있다.
* 업무 기록에 태그, 프로젝트, 관련 문서를 연결할 수 있다.
* 사용자가 개인 To-do를 작성하고 완료 상태를 관리할 수 있다.
* 업무 기록과 To-do를 AI 문서 생성의 참고 자료로 선택할 수 있다.
* 업무 다이어리 기반으로 주간/월간 보고서 초안을 생성할 수 있다.
* 개인 기록과 팀 공유 기록을 구분할 수 있다.
* 권한과 공개 범위에 따라 업무 맥락 데이터 접근이 제한된다.

# 5. 주요 사용자 흐름

## 5.1 업무 다이어리 작성

### 기본 흐름

1. 사용자가 업무 다이어리 화면에 진입한다.
2. 날짜를 선택한다.
3. 업무 내용을 입력한다.
4. 태그 또는 프로젝트를 선택한다.
5. 관련 문서가 있으면 연결한다.
6. 공개 범위를 선택한다.
7. 저장한다.

### 예외 흐름

* 내용 없이 저장하려 하면 입력 안내를 표시한다.
* 권한 없는 문서는 관련 문서로 연결할 수 없다.
* 팀 공개 기록은 팀 소속 사용자만 접근할 수 있다.
* 삭제된 관련 문서는 연결 해제 또는 비활성 상태로 표시한다.

---

## 5.2 업무 다이어리 조회

### 기본 흐름

1. 사용자가 업무 다이어리 화면에 진입한다.
2. 오늘 기록을 기본으로 확인한다.
3. 날짜, 태그, 프로젝트로 필터링한다.
4. 특정 기록을 선택해 상세를 확인한다.
5. 권한이 있으면 수정하거나 삭제한다.

### 예외 흐름

* 기록이 없으면 빈 상태와 작성 버튼을 표시한다.
* 다른 사용자의 개인 기록은 조회할 수 없다.
* 팀 공개 기록은 팀 권한에 따라 조회 가능하다.

---

## 5.3 To-do 작성 및 완료 처리

### 기본 흐름

1. 사용자가 To-do 화면에 진입한다.
2. 새 To-do를 작성한다.
3. 담당자, 마감일, 우선순위, 관련 문서를 설정한다.
4. 진행 상태를 변경한다.
5. 완료 시 완료 처리한다.

### 예외 흐름

* 담당자를 지정하지 않으면 본인을 기본 담당자로 설정한다.
* 마감일 없이 저장할 수 있다.
* 권한 없는 사용자를 담당자로 지정할 수 없다.
* 완료된 To-do는 보고서 생성 시 완료 업무로 분류된다.

---

## 5.4 업무 기록 기반 보고서 생성

### 기본 흐름

1. 사용자가 업무 다이어리 또는 AI 문서 생성 화면에서 보고서 생성을 선택한다.
2. 기간을 선택한다.
3. 포함할 업무 기록과 To-do를 선택한다.
4. 문서 유형을 업무 보고서로 설정한다.
5. AI가 선택된 업무 맥락을 참고해 보고서 초안을 생성한다.
6. 사용자가 결과를 미리보고 편집기로 이동한다.
7. 문서로 저장한다.

### 예외 흐름

* 선택 기간에 업무 기록이 없으면 직접 입력 또는 빈 보고서 생성을 안내한다.
* 권한 없는 팀 기록은 선택 목록에 표시하지 않는다.
* 너무 많은 기록을 선택하면 요약 또는 개수 제한을 적용한다.

---

## 5.5 문서 생성 시 업무 맥락 선택

### 기본 흐름

1. 사용자가 AI 문서 생성 화면에서 참고 자료 선택 영역을 연다.
2. 문서, 업무 기록, To-do 탭을 확인한다.
3. 필요한 업무 기록과 To-do를 선택한다.
4. AI 생성 요청에 선택된 업무 맥락이 포함된다.
5. 생성 결과에서 참고된 업무 맥락 출처를 확인한다.

### 설계 기준

* 선택 가능한 자료는 접근 권한이 있는 항목으로 제한한다.
* 개인 기록은 작성자 본인만 선택할 수 있다.
* 팀 공개 기록은 팀 권한에 따라 선택할 수 있다.
* 선택된 업무 맥락은 생성 결과의 출처로 표시한다.

AI 생성에 업무 맥락을 사용할 때는 다음 원칙을 따른다.

* 사용자가 접근 가능한 DiaryEntry만 선택할 수 있다.
* 사용자가 접근 가능한 Todo만 선택할 수 있다.
* 개인 기록은 작성자 본인이 명시적으로 선택해야 한다.
* 팀 공개 기록은 팀 구성원만 선택할 수 있다.
* AI 결과에는 참고된 업무 기록과 To-do 출처가 표시된다.

# 10. 업무 맥락 품질 기준

## 10.1 좋은 업무 기록 기준

업무 기록은 이후 문서 생성에 활용되므로 지나치게 짧거나 모호하지 않게 작성할 수 있도록 돕는다.

### 권장 요소

* 수행한 일
* 관련 대상
* 결과
* 이슈
* 다음 행동

---

## 10.2 입력 도움 UX

### 예시 문구

* “A 고객사 미팅 진행, 신규 제안 범위 논의”
* “결제 API 오류 수정 완료, QA 재검증 필요”
* “디자인 수정 요청 접수, 금요일까지 1차 시안 전달 예정”

---

## 10.3 보고서 생성 품질 기준

* 완료 업무와 미완료 업무를 구분한다.
* 날짜와 프로젝트 맥락을 반영한다.
* 사용자가 기록하지 않은 내용을 과도하게 추정하지 않는다.
* 불확실한 내용은 확인 필요 항목으로 표시한다.
* 팀 공유 문서로 사용할 수 있는 문체를 유지한다.

---

# 11. 오류 및 예외 처리

## 11.1 업무 기록 오류

* 내용 없는 저장
* 권한 없는 문서 연결
* 삭제된 관련 문서 연결
* 다른 조직 프로젝트 연결

### UX 기준

* 저장 실패 시 입력 내용을 유지한다.
* 연결할 수 없는 항목은 선택 목록에서 제외한다.
* 삭제된 연결 항목은 비활성 상태로 표시한다.

---

## 11.2 To-do 오류

* 존재하지 않는 담당자 지정
* 권한 없는 담당자 지정
* 잘못된 마감일
* 삭제된 문서 연결

---

## 11.3 AI 참고 자료 오류

* 선택된 업무 기록이 삭제됨
* 선택된 To-do 권한이 변경됨
* 선택 자료가 너무 많음
* 선택 기간에 기록이 없음

### 대응 기준

* 생성 직전에 권한과 상태를 다시 확인한다.
* 사용할 수 없는 자료는 제외하고 사용자에게 표시한다.
* 자료가 부족하면 추가 입력을 안내한다.

---

# 12. 테스트 계획

## 12.1 핵심 테스트 시나리오

| 시나리오 | 검증 내용 |
| --- | --- |
| 업무 기록 작성 | 날짜, 내용, 공개 범위 저장 |
| 업무 기록 조회 | 날짜별 기록 표시 |
| 업무 기록 수정 | 수정 내용 반영 |
| 업무 기록 삭제 | 일반 목록에서 제외 |
| 관련 문서 연결 | 접근 가능한 문서 연결 |
| To-do 작성 | 제목, 담당자, 마감일 저장 |
| To-do 완료 처리 | 완료 상태 반영 |
| 기간별 보고서 생성 | 선택 기간 기록 기반 AI 생성 |
| 업무 기록 참고 생성 | AIReference에 DiaryEntry 연결 |
| To-do 참고 생성 | AIReference에 Todo 연결 |

---

## 12.2 권한 테스트

* 다른 사용자의 Private 업무 기록을 조회할 수 없는지 확인
* 같은 팀 사용자가 Team 업무 기록을 조회할 수 있는지 확인
* 다른 팀 사용자가 Team 업무 기록을 조회할 수 없는지 확인
* 권한 없는 문서를 업무 기록에 연결할 수 없는지 확인
* AI 생성 시 권한 없는 업무 기록이 선택되지 않는지 확인
* To-do 담당자가 아닌 사용자의 접근 범위가 제한되는지 확인

---

## 12.3 AI 연결 테스트

* 업무 기록을 선택해 보고서가 생성되는지 확인
* 완료 To-do와 미완료 To-do가 구분되어 반영되는지 확인
* 생성 결과에 출처가 표시되는지 확인
* 업무 기록이 없는 기간에서 적절한 안내가 표시되는지 확인
* 생성 직전 권한이 변경된 자료가 제외되는지 확인

---

# 13. 완료 기준

E단계는 다음 기준을 만족하면 완료로 본다.

* Editor 이상 사용자가 업무 다이어리를 작성할 수 있다.
* 사용자가 날짜별 업무 기록을 조회, 수정, 삭제할 수 있다.
* 업무 기록에 태그, 프로젝트, 관련 문서를 연결할 수 있다.
* 사용자가 To-do를 작성하고 완료 상태를 관리할 수 있다.
* 업무 기록과 To-do에 공개 범위가 적용된다.
* AI 문서 생성에서 업무 기록과 To-do를 참고 자료로 선택할 수 있다.
* 업무 기록 기반 보고서 초안을 생성할 수 있다.
* AI 결과에 사용된 업무 맥락 출처가 표시된다.
* 권한 없는 업무 기록과 To-do는 조회 또는 참고 자료 선택이 차단된다.

# 14. 주요 리스크 및 대응

| 리스크 | 설명 | 대응 방향 |
| --- | --- | --- |

| To-do 범위 확대 | 업무 관리 도구처럼 커질 수 있음 | MVP에서는 보고서 생성에 필요한 최소 기능으로 제한 |
| 공개 범위 혼란 | 개인 기록과 팀 공유 기록이 섞일 수 있음 | 기본 공개 범위와 표시를 명확히 구분 |
| AI 맥락 과다 | 너무 많은 기록을 넣으면 결과 품질 저하 가능 | 기간, 개수, 요약 기준 제한 |
| 권한 누락 | 개인 기록이 잘못 노출될 수 있음 | 조회, 선택, 생성 직전 권한 검사 |
| 프로젝트 기능 비대화 | 프로젝트 관리 기능으로 범위가 커질 수 있음 | 초기에는 분류 단위로만 제공 |

# 15. F단계 연결

E단계 완료 후 F단계 사내 지식 검색 구현으로 넘어간다.

F단계 착수 전 준비되어야 할 항목은 다음과 같다.

* 업무 기록 데이터
* To-do 데이터
* 태그와 프로젝트 연결
* 관련 문서 연결
* 업무 맥락 접근 권한
* AIReference에 업무 기록과 To-do 연결
* 문서 생성 결과의 출처 표시 구조
