# 09. E단계 업무 맥락 계획서 작업 보고서

## 대상 문서

`docs/implementation/notive-implementation-plan-e-work-context-v1.0.md`

E단계(업무 맥락)에서 무엇을 어떻게 만들지 정의한 세부 계획서다. 업무 다이어리, To-do, 그리고 이런 업무 기록을 AI가 문서 생성에 참고하는 흐름이 여기서 다뤄진다.

## 변경한 내용

### 1. §7.4 Todo 데이터 모델을 개인 전용으로 정정

기존 주요 필드: 담당자 ID, 생성자 ID, 팀 ID, 마감일, 우선순위, 상태, 공개 범위, 관련 문서 ID, 관련 프로젝트 ID.

기존 상태값: NotStarted, InProgress, Done, Cancelled.

변경 후 주요 필드: **소유자 ID(생성자 = 담당자, 단일 사용자)**, 마감일, 우선순위, 상태, 관련 문서 ID, 관련 프로젝트 ID.

변경 후 상태값: `todo`, `in_progress`, `done`.

문서 위에 메모 추가: "MVP는 개인 To-do만 지원한다(Phase A §15). 담당자/팀/공개 범위/Cancelled 상태는 후속 단계로 미룬다."

### 2. §8.2 To-do 권한 표를 개인 전용으로 축소

기존 표:

| 조건 | 접근 기준 |
| --- | --- |
| 개인 To-do | 담당자와 생성자 |
| 팀 To-do | 같은 팀 구성원 |
| 문서 연결 To-do | 연결 문서 접근 권한 필요 |
| 관리자 조회 | 정책에 따라 팀 또는 조직 범위 |

변경 후:

| 조건 | 접근 기준 |
| --- | --- |
| 개인 To-do | 생성자 본인만 |
| 문서 연결 To-do | 생성자 + 연결 문서 접근 권한 필요 |

관리자 조회와 팀 To-do 행 제거.

### 3. §8.3 AI 참고 권한 정정

기존: "사용자가 접근 가능한 Todo만 선택할 수 있다." + "팀 공개 기록은 팀 구성원만 선택할 수 있다."

변경: "Todo는 본인이 생성한 것만 선택할 수 있다(MVP는 개인 전용)." + DiaryEntry에 대한 팀 공개 규칙은 유지(다이어리는 팀 공유가 가능하므로).

## 변경 이유

- **Phase A §15 정합**: To-do를 개인 전용으로 잠갔는데 E단계 데이터 모델에 담당자/팀/공개 범위가 그대로 남아 있으면 개발자가 협업 기능을 만들었다가 다시 떼어내는 헛수고를 한다.
- **상태값 표기 통일**: 보안 정책, DB 설계, API 명세에서 모두 소문자(`todo`/`in_progress`/`done`)를 쓰는데 이 문서만 카멜케이스(NotStarted/InProgress/Done/Cancelled)면 변환 로직이 곳곳에 생긴다. 한 가지 표기로 통일.
- **DiaryEntry는 손대지 않은 이유**: Phase A §15에서 업무 다이어리는 Private/Team 두 공개 범위를 유지했다(개인 전용으로 좁히지 않았음). 따라서 §8.3의 다이어리 관련 규칙은 그대로 유지.

## B단계 착수에 미치는 영향

- E단계는 B단계로부터 세 단계 떨어져 있지만, **B단계 권한 모듈 설계에 영향**을 준다. 권한 모듈이 To-do에 대해 처리해야 할 케이스가 줄어들어(생성자 한 명만 검사하면 됨) 코드가 단순해진다.
- DB 설계서의 todos 테이블도 같은 방향으로 단순화돼 있어, B단계 마이그레이션과 E단계 마이그레이션이 같은 가정 위에 놓인다.

## 남은 주의사항

- E단계의 "업무 다이어리 기반 보고서 생성"(AI가 다이어리를 참고해 보고서를 만드는 흐름)은 권한 검사가 까다롭다. 다이어리 작성자와 보고서 요청자가 다른 사람일 때, 다이어리의 공개 범위를 정확히 따라야 한다. E단계 진입 시 §8.3 규칙을 권한 모듈 테스트 케이스로 반드시 변환해야 한다.
- To-do가 개인 전용이지만 향후 팀 협업 To-do를 도입할 때 어떤 마이그레이션이 필요한지(추가될 칸 목록)는 DB 설계서 §10.3에 메모로 남아 있다. 후속 단계에서 그 메모를 출발점으로 삼는다.
- "개인 전용"이라는 결정이 사용자에게 도움이 될지, 아니면 팀 단위 To-do가 필요해 다른 도구로 사용자가 옮겨갈지는 출시 후 피드백으로 검증해야 한다.
