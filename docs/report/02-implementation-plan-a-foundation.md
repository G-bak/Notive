### 2. MVP에서 결정한 12개 항목을 §15에 잠금

기존에는 "이런 선택지가 있다"고만 적혀 있던 항목들을 명확히 결정해 표로 잠갔다.

| 결정한 항목 | 결정 내용 |
| --- | --- |
| 가입 방식 | 누구나 이메일로 가입 가능. 가입 직후 초대 수락 또는 새 조직 생성 둘 중 하나를 선택해야 한다. 이메일 인증 필수. |
| 최초 Admin | 새 조직을 만든 사용자가 자동으로 그 조직의 첫 Admin이 된다. 조직에 Admin이 한 명만 남으면 강등/비활성화가 막힌다. |
| 한 사용자가 여러 조직? | MVP는 1인 1조직. 여러 조직 동시 소속은 미루기. |
| 팀 구조 | 한 단계만(상위 팀/하위 팀 없음). 부서는 팀과 같은 개념으로 통합. **한 사용자는 한 팀에만 소속**. |
| 문서 기본 공개 범위 | 작성자만 볼 수 있는 Private. 사용자가 명시적으로 넓힐 때만 팀/조직 공개로 바뀐다. |
| Admin이 문서 본문을 볼 수 있는가 | 묵시적으로는 못 본다. Admin은 메타데이터(제목, 소유자, 시각 등)만 본다. 본문 보려면 일반 권한 규칙을 거쳐야 한다. |
| 업무일지 공개 | 기본 Private, 본인이 원하면 팀 공개. 조직 공개는 미루기. |
| To-do 범위 | 개인 전용. 다른 사람에게 할당하거나 팀 To-do는 미루기. 상태값은 todo / in_progress / done 세 가지. |
| 문서 삭제 정책 | 즉시 영구 삭제하지 않고 휴지통에 30일 보관. 30일이 지나면 자동 삭제. 그 안에는 복구 가능. |
| AI 로그 보관 | 기본은 메타데이터만 90일. AI에게 보낸 요청 본문, 받은 응답 본문은 영구 저장하지 않는다. 사용자가 "문제 신고"를 누른 경우에만 본문 30일 보관. |
| 인증 방식 | 이메일 + 비밀번호 + 이메일 인증. 비밀번호는 최소 10자, 유출 비밀번호 차단. SSO와 2단계 인증은 후속. |
| Phase B 최소 데이터 | User, Organization, Membership, Team, Role, Invitation, Session, AuditLog 골격까지. 문서/AI 테이블은 B단계에서 안 만든다. |

### 3. 다중 팀 소속 결정 강하

처음에는 "한 사용자가 여러 팀에 속할 수 있다"고 적었지만, 실제 DB 설계는 한 사람당 한 팀만 저장하도록 되어 있어 충돌이 났다. **"한 사용자 = 한 팀"으로 결정을 낮추고**, 다중 팀은 후속 단계로 넘겼다.

### 4. §16 Deferred(미루기) 항목과 §17 Phase B 진입 기준 추가

MVP에 넣지 않을 항목들(다중 조직, 외부 링크 공유, SSO, 2FA, 모바일 앱 등)을 명시적으로 적어 두어 B단계 이후에 슬그머니 들어오는 것을 막는다.

§17은 "다음 조건이 모두 만족돼야 B단계를 시작할 수 있다"는 체크리스트다.

## 변경 이유

- **결정을 잠근 이유**: B단계(서비스 기반 구축)에 들어가려면 가입 방식, 인증, 권한 모델 같은 핵심 결정이 미리 확정되어 있어야 한다. 그렇지 않으면 코드 다 짜놓고 결정이 바뀌어 다시 짜게 된다.
- **다중 팀 → 단일 팀**: DB가 단일 팀을 가정하고 설계되어 있는데 계획서가 다중 팀이라고 적혀 있으면 둘 중 하나는 거짓말이다. 다중 팀을 진짜로 지원하려면 별도 테이블과 권한 쿼리 재설계가 필요한데, MVP 일정에서 그만큼 가치가 없다고 판단했다.
- **AI 본문 미저장**: AI에게 사용자가 보낸 요청과 받은 응답에는 회사 내부 정보가 그대로 들어갈 수 있다. 운영자가 실수로 보거나 데이터가 유출되면 큰 문제다. 그래서 기본은 저장 안 함, 사용자가 명시적으로 신고할 때만 30일만 보관하는 것으로 보수적으로 잡았다.

## B단계 착수에 미치는 영향

- B단계가 만들 테이블 목록이 §15에 명확히 적혀 있어 작업 범위가 분명해졌다.
- 가입/인증/Admin 보호 로직이 세부까지 잠겨 있어 코드 작성 시 판단을 다시 할 필요가 없다.
- "한 사용자 = 한 팀" 결정으로 권한 쿼리 코드가 단순해진다(여러 팀 합치는 쿼리를 짤 필요 없음).
- 가장 중요한 점: §17 진입 기준을 통과해야 B단계를 시작한다. 이번 작업으로 그 기준이 만족됨.

## 남은 주의사항

- §15.1 "잔여 미결정"에 아직 정해지지 않은 항목이 있다(Manager의 사용자 초대 권한, Manager의 템플릿 관리, 권한 오류 코드 NOT_FOUND vs FORBIDDEN). 이 항목들은 B단계 진행 중 마주칠 때 다시 결정해야 한다.
- AI 본문을 단기적으로 보관하는 "세션 바운드 단기 스토리지"의 실제 인프라(Redis 등)와 운영 방식은 B단계 인프라 결정에서 마무리되어야 한다.
- 모든 결정의 근거가 이 문서에 적혀 있으므로, B단계 이후 어떤 결정이 흔들릴 때마다 가장 먼저 이 §15를 다시 보고 정합을 맞춰야 한다.
