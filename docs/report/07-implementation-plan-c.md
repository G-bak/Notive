# 07. C단계 문서 관리 계획서 작업 보고서

## 대상 문서

`docs/implementation/notive-implementation-plan-c-document-management-v1.0.md`

C단계(문서 관리)에서 무엇을 어떻게 만들지 정의한 세부 계획서다. 문서 목록, 편집, 저장, 공유, 버전 관리 같은 핵심 기능이 여기서 다뤄진다.

## 변경한 내용

### 1. §8.3 문서 공유 범위 표에서 Department 행 삭제

기존:

| 공유 범위 | 접근 대상 |
| --- | --- |
| Private | 작성자 또는 소유자 |
| Team | 지정 팀 사용자 |
| Department | 지정 부서 사용자 |
| Organization | 조직 전체 사용자 |
| Specific Users | 지정 사용자 |

변경: Department 행을 삭제. 표 아래에 "MVP는 Department 공유 범위를 두지 않고 Team으로 통합한다(Phase A §15)" 메모 추가.

### 2. §8.4 Admin 권한 표현 정정

기존: "Admin의 전체 문서 접근 허용 여부는 A단계 미결정 사항에 따라 적용한다."

변경: "Admin은 조직 전체 문서 본문에 대한 묵시적 접근을 가지지 않는다(Phase A §15). 메타데이터만 조회하며, 본문은 일반 권한 규칙(Organization 공개 또는 명시 공유)을 따른다."

## 변경 이유

- **Department 제거**: Phase A §15와 보안 정책에서 Department를 Team으로 통합하기로 잠갔다. C단계 계획서에 그대로 남아 있으면 개발자가 5종류 공유 범위를 만들었다가 다시 4종류로 줄이는 헛수고를 한다.
- **Admin 권한 표현 정정**: "미결정 사항에 따라 적용"이라는 모호한 표현이 그대로 남아 있으면 개발 시 결정이 다시 필요하다고 오해할 수 있다. 결정이 내려졌으니 그 결정을 명시했다.

## B단계 착수에 미치는 영향

- C단계는 B단계 다음이지만, **B단계 권한 모듈이 처리해야 할 공유 범위가 4종류로 확정**되어 권한 검사 코드를 한 번만 짜면 된다.
- B단계 후 C단계 진입 시 이 문서가 곧장 작업 기준으로 사용된다.

## 남은 주의사항

- C단계의 문서 본문 저장 형식(Markdown vs JSON document vs HTML)은 아직 미결정이다(DB 설계 §16.1 참조). C단계 시작 전에 결정이 필요하다.
- 자동 저장 주기(Phase A는 10초로 잠금)가 C단계 구현에 그대로 반영됐는지 C단계 진입 시 점검이 필요하다.
- 향후 Department를 별도 개념으로 분리해야 할 경우, 이 §8.3 표와 보안 정책 §6.1을 함께 변경해야 한다(두 문서가 한 짝이다).
