# 01. 전체 구현 계획서 작업 보고서

## 대상 문서

`docs/implementation/notive-implementation-plan-v1.0.md`

Notive 제품을 처음 만들 때 어떤 단계로(Phase A → H) 무엇을 만들지 큰 그림으로 적어둔 문서다. 실제 코드를 어떻게 짤지가 아니라 "무엇을 언제 만들 것인가"의 지도다.

## 변경한 내용

1. 문서 전체를 한국어에서 영어로 다시 썼다.
2. "팀 / 부서(Team / department)"라고 두 단어가 같이 쓰이던 부분을 모두 "팀(Team)"으로 통일했다.
3. 데이터 영역 표에서 "Team" 항목 설명을 "Department는 MVP에서 Team으로 통합한다"로 바꿨다.

## 변경 이유

- **영어로 정리한 이유**: 협업 도구(Claude·Codex)와 외부 자료가 영어 기준으로 동작하므로, 모두가 같은 단어로 같은 개념을 가리킬 수 있게 하기 위해서다. 사용자 화면에 보이는 글이 아니라 내부 작업용 문서이므로 영어가 더 효율적이다.
- **부서/팀 통일**: MVP에서는 부서(Department)와 팀(Team)을 따로 관리하지 않기로 결정했기 때문이다. 두 개념을 따로 두면 권한 관리가 복잡해지는데, 초기 고객에게는 한 가지 단위만 있으면 충분하다. Phase A의 §15 결정과 정합을 맞춘 것이다.

## B단계 착수에 미치는 영향

- B단계는 이 영문 계획서를 기준으로 작업을 시작한다. 단어 통일이 끝났기 때문에 "팀이라는 게 부서랑 같은 거야?" 같은 질문이 나오지 않는다.
- 데이터베이스에 Team 한 종류만 만들면 되므로 B단계의 권한 모듈과 화면 설계가 단순해진다.

## 남은 주의사항

- 향후 엔터프라이즈 고객이 부서(Department) 개념을 별도로 요구할 경우, 이 문서에 명시된 "Team으로 통합" 결정을 다시 열어야 한다. 그때는 권한 정책과 DB 설계까지 함께 변경된다.
- 영문화 과정에서 의역한 표현이 있으므로, 한국어 원문 의도와 미세한 차이가 생긴 부분은 향후 리뷰에서 확인이 필요하다.
