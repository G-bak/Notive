# 05. AI 생성 정책 작업 보고서

## 대상 문서

`docs/ai/notive-ai-generation-policy-v1.0.md`

AI가 어떻게 문서를 만들어야 하는지(어떤 출처를 쓰고, 무엇을 절대 하지 말아야 하고, 결과를 어떻게 다룰지)를 정의하는 정책 문서다. 실제 프롬프트 문안이 아니라 원칙 모음이다.

## 변경한 내용

### §12 로그 및 보존 정책을 재작성

기존에는 "이런 데이터를 기록할 수 있다", "민감 정보는 최소 범위만 저장한다" 같은 모호한 표현으로 적혀 있었다.

이번에 5개 소절로 명확히 잠갔다.

- **§12.1 기본 저장 (메타데이터만)**: 누가, 언제, 어떤 문서 유형으로, 어떤 템플릿을 골라, 몇 개의 참고 자료 ID를 사용해서, 성공/실패가 어떻게 나왔고, 시간이 얼마나 걸렸고, 토큰을 얼마 썼는지를 90일간 보관.
- **§12.2 기본 미저장 항목**: AI에게 보낸 요청 본문, AI가 만든 결과 본문, 참고 자료 본문, 자유 형식 오류 메시지는 영구 테이블에 안 적는다. 미리보기/편집 중인 본문은 단기 스토리지에 잠깐 보관, 미저장 24시간 또는 폐기 시 즉시 삭제.
- **§12.3 opt-in 문제 신고 페이로드**: 사용자가 직접 "문제 신고"를 누른 경우에 한해 그 요청의 본문을 30일 보관. 신고자 ID와 신고 사유 코드 동반 저장.
- **§12.4 관리자 노출**: Admin 화면에서도 본문은 안 보인다. 메타데이터만.
- **§12.5 정리 작업**: 90일/30일이 지나면 백그라운드 작업이 자동 삭제.

### §15 미결정 표 정리

Phase A에서 결정된 항목(AI 요청 원문 저장, AI 결과 본문 보존, 관리자 AI 로그 조회)을 미결정 표에서 빼고 결정 표로 옮겼다. 진짜 미결정인 항목(조직별 문체, 재생성 횟수 제한, 프롬프트 관리 방식)만 §15.1에 남겼다.

## 변경 이유

- **모호한 정책 → 숫자로 잠금**: "최소 범위만 저장한다" 같은 문장은 사람마다 해석이 다르다. 운영자가 "이건 최소 범위야"라며 본문을 저장해 버리면 사고가 난다. 그래서 **무엇을 저장하고 무엇을 안 저장하는지 항목별로 못박았다**.
- **본문 미저장 원칙**: AI 요청과 결과에는 회사 내부 정보(고객 이름, 금액, 인사 결정 등)가 그대로 담길 수 있다. 영구 저장하면 사고 시 피해가 너무 크다.
- **그래도 30일 신고 페이로드를 둔 이유**: AI 품질 개선과 사고 분석을 위해 최소한의 데이터는 필요하다. 사용자가 명시적으로 동의한(신고 버튼을 누른) 단일 요청만 보관하면, 무차별 수집 없이도 핵심 사례를 분석할 수 있다.

## B단계 착수에 미치는 영향

- AI 기능 구현은 D단계에서 본격적으로 진행되지만, **B단계에서 만드는 인프라(단기 스토리지, 백그라운드 작업 시스템)**가 이 정책을 지원해야 한다.
- "본문은 단기 스토리지에 보관"이라는 정책 때문에 B단계 인프라 결정에 캐시/세션 스토리지(Redis 등)가 포함된다.
- 90일/30일 자동 정리 작업을 돌릴 백그라운드 작업 인프라(예: cron, worker queue)도 B단계에서 결정해야 한다.

## 남은 주의사항

- 본문이 영구 저장되지 않으므로 **AI 품질 회귀 분석이 어려워진다**. opt-in 신고만으로 충분한 데이터를 모을 수 있는지 운영 시작 후 검증이 필요하다.
- §15.1 잔여 미결정(조직별 문체, 재생성 횟수 제한, 프롬프트 관리 방식)은 D단계에서 다시 결정해야 한다.
- 단기 스토리지의 24시간 만료 정책은 정책 문서로는 잠겼지만, **실제 인프라 수준의 자동 만료 처리(TTL 설정, 정리 작업)** 검증은 D단계 구현 후 QA에서 확인이 필요하다.
- "참고 자료 본문 스냅샷"을 저장하지 않는 결정 때문에, AI가 어떤 자료를 어떻게 참고했는지 사후 추적이 제한된다. 출처 ID는 남으므로 사후 자료 본문을 다시 가져올 수는 있지만, 그 사이 자료가 수정/삭제되면 당시 모습을 재현할 수 없다는 한계가 있다.
