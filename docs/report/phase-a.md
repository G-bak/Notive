# Phase A 전체 작업 보고서

## 1. 보고 범위

본 보고서는 Notive 구현 단계 중 **Phase A — 기반 설계(Foundation Design)** 가 완료된 시점에 작성된, 작업 전체를 한 번에 정리하는 보고서다. 비개발자도 어떤 작업이 왜 이루어졌는지를 따라갈 수 있도록 풀어 썼고, 각 결정의 근거와 후속 영향을 함께 적었다.

이후 단계(B–H)는 별도 보고서로 다룬다.

---

## 2. Phase A는 무엇이고, 왜 했는가

Phase A는 코드 한 줄 짜기 전에 "무엇을 만들지", "누가 무엇을 할 수 있게 할지", "어떤 데이터를 어떤 규칙으로 보관할지"를 확정하는 단계다.

이 단계의 결과물이 흔들리면 이후 단계가 모두 흔들리기 때문에 다음 두 가지를 가장 중요하게 다뤘다.

* **결정을 명확히 잠근다.** 모호한 표현은 구현자가 알아서 해석하게 두지 않고, "이렇게 한다"로 못박는다.
* **모든 관련 문서가 같은 결정을 가리키게 한다.** 보안 정책, DB 설계, API 명세, AI 정책처럼 서로 영향을 주고받는 문서들이 충돌하면 다음 단계가 시작될 수 없다.

## 5. MVP 결정 12개 — Phase A §15 Locked Decisions

Phase A 작업의 핵심이다. "이렇게 한다"로 못박은 12개 항목은 다음과 같다. 각 항목 옆에 결정과 그렇게 한 이유를 함께 적었다.

### 5.1 가입 방식 — 셀프 가입 + 이메일 인증 필수

**결정**: 누구나 이메일과 비밀번호로 가입할 수 있다. 다만 이메일 인증 링크를 클릭해야 계정이 활성화된다. 인증 후에는 **새 조직을 만들거나, 받은 초대를 수락하거나** 둘 중 하나를 골라야 다음 화면으로 진행된다. "기존 조직에 마음대로 가입 신청"하는 흐름은 없다.

**이유**: 회사 내부 데이터를 다루기 때문에 무차별 가입을 막아야 하지만, 초기 부트스트랩(첫 조직을 만드는 사람)이 가능해야 한다. 이메일 인증을 필수로 두면 가입 자체는 열어두고 데이터 접근은 통제할 수 있다.

### 5.2 최초 Admin — 조직 생성자 자동 부여 + Last-Admin 보호

**결정**: 새 조직을 만든 사람이 자동으로 그 조직의 첫 Admin이 된다. 별도 승인 절차 없음. 단, 조직에 활성 Admin이 한 명만 남으면 그 사람을 강등하거나 비활성화하거나 조직에서 빼거나 다른 조직으로 옮길 수 없다(네 가지 동작 모두 차단).

**이유**: MVP에 별도 승인 시스템은 과하다. 자동 부여가 가장 단순하다. 다만 Admin이 0명이 되면 조직 운영이 막혀 복구가 어렵기 때문에 마지막 Admin은 보호해야 한다.

### 5.3 한 사용자 = 한 조직

**결정**: MVP에서 한 사용자는 정확히 한 조직에 속한다. 여러 조직 동시 소속은 후속 단계로 미룬다.

**이유**: 다중 조직 소속은 사용자 화면(조직 전환 UI)과 권한 검사(어느 조직 컨텍스트인지) 모두에 복잡도를 더한다. 초기 고객의 실제 요구를 보고 도입해도 늦지 않다.

### 5.4 한 사용자 = 한 팀(primary team)

**결정**: 사용자는 자기 조직 안에서 정확히 한 팀에만 속한다. 다중 팀 소속은 후속 단계로 미룬다. 팀 구조는 단일 레벨(상위 팀/하위 팀 없음)이고, "부서"는 별도 개념이 아니라 "팀"으로 통합한다.

**이유**: 처음에 다중 팀을 허용하려고 했지만, 데이터베이스 설계가 단일 팀 컬럼을 가정하고 있어 충돌이 났다. 다중 팀을 진짜 지원하려면 별도 조인 테이블과 권한 쿼리 재설계가 필요한데, MVP 일정에서 그만큼 가치가 없다고 판단해 단순한 쪽으로 강하했다.

### 5.5 문서 기본 공개 범위 — Private

**결정**: 새 문서의 기본 공개 범위는 작성자만 볼 수 있는 Private이다. 사용자가 명시적으로 넓힐 때만 Team 공개 또는 Organization 공개로 바뀐다. MVP에서 사용 가능한 공개 범위는 Private / Team / Organization / Specific Users 네 가지이고, "Department 공개"는 두지 않는다.

**이유**: 회사 내부 자료가 무심코 공개되는 사고를 막기 위해 보수적인 기본값을 잡았다. 의도한 사람만 보게 하려면 기본은 잠겨 있어야 한다.

### 5.6 Admin 문서 본문 열람 — 메타데이터만

**결정**: Admin은 조직 전체 문서의 **메타데이터(제목, 소유자, 공유 범위, 크기, 시각)** 는 모두 볼 수 있지만, **본문**은 일반 권한 규칙을 따라야 본다. 즉, 문서가 Organization 공개거나 Admin에게 직접 공유된 경우에만 Admin이 본문을 본다. "관리자니까 다 볼 수 있다"는 묵시적 권한은 MVP에 없다.

**이유**: Notive는 회사 내부 문서를 다룬다. 작성자가 "내가 쓴 글이 누구에게 노출되는가"를 정확히 예측할 수 있어야 신뢰가 생긴다. 컴플라이언스(법적 자료 제출 같은 특수 상황)에 대응하려면 별도 "비상 접근" 절차가 필요한데, 이건 후속 단계에서 별도 설계한다.

### 5.7 업무일지 공개 — 기본 Private + Team 공유 가능

**결정**: 업무 다이어리(개인 일일 업무 기록)의 기본 공개 범위는 Private(작성자만)이다. 작성자가 원하면 Team으로 공유할 수 있다. Organization 공개는 후속 단계로 미룬다. AI가 다이어리를 참고 자료로 쓸 수 있는 조건은 일반 권한 규칙과 같다(요청자가 그 다이어리를 볼 권한이 있어야 함).

**이유**: 다이어리는 개인의 업무 메모 성격이라 기본은 잠겨 있어야 한다. 팀 공유는 보고용으로 자주 쓰이므로 옵트인 형태로 허용한다.

### 5.8 To-do — 개인 전용

**결정**: To-do는 본인이 만든 것만 본인만 본다. 다른 사람에게 할당하는 기능, 팀 To-do 기능, Manager/Admin이 다른 사람의 To-do를 보는 기능은 모두 후속 단계로 미룬다. 상태값은 `todo`, `in_progress`, `done` 세 가지만 쓴다.

**이유**: 팀 협업 To-do는 할당, 알림, 권한 다툼 같은 부수 기능이 따라온다. MVP에서 필요한 가치보다 만드는 비용이 크다.

### 5.9 문서 삭제 정책 — 소프트 삭제 + 30일 휴지통

**결정**: 문서를 삭제하면 즉시 영구 삭제되지 않고 휴지통에 30일 보관된다. 그 안에는 소유자나 Admin이 복구할 수 있다. 30일이 지나면 백그라운드 작업이 자동으로 영구 삭제한다. 30일 안에 영구 삭제하려면 Admin의 명시적 "purge" 동작이 필요하다.

**이유**: 사용자 실수로 삭제된 문서를 되돌릴 수 있어야 데이터 신뢰가 유지된다. 30일은 사용자가 실수를 알아차리기에 충분한 기간이고, 무한 보관은 저장 비용과 개인정보 관점에서 부적절하다.

### 5.10 AI 로그 보관 — 메타데이터만 90일, 본문은 신고 시 30일

**결정**: AI 요청 데이터베이스에는 **메타데이터만** 영구 저장한다(누가, 언제, 어떤 문서 유형으로, 토큰을 얼마 썼고, 결과를 저장했는지 등). **요청 본문(사용자가 AI에게 보낸 글)과 응답 본문(AI가 만든 글)은 영구 저장하지 않는다**. 미리보기/편집 중인 본문은 단기 스토리지에 24시간만 보관되고, 사용자가 저장하면 일반 문서로 옮겨지고 그렇지 않으면 자동 삭제된다.

사용자가 명시적으로 "문제 신고"를 누른 경우에만 그 단일 요청의 본문이 별도 테이블(`ai_request_payloads`)에 30일 한도로 보관된다. 보관 사유 코드와 신고자 정보가 함께 저장된다.

표준 메타데이터 보존 기간: **90일**. 신고 본문 보존 기간: **30일**.

**이유**: AI 요청과 결과에는 회사 내부 정보(고객 이름, 금액, 인사 결정 등)가 그대로 담길 수 있다. 영구 저장하면 운영자 실수, 정책 변경, 데이터 유출 사고 시 피해가 너무 크다. 그렇다고 모든 데이터를 안 남기면 품질 문제 분석이 불가능하므로, **사용자가 동의한 단일 요청만 단기 보관**하는 균형을 잡았다.

### 5.11 인증 방식 — 이메일 + 비밀번호 + 이메일 인증 + 서버 세션

**결정**: 이메일 + 비밀번호 인증, 이메일 인증 필수, 서버 측 세션 저장(PostgreSQL `sessions` 테이블; 토큰은 해시로 보관). 비밀번호 정책: 최소 10자, 영문/숫자 혼합, 가입과 변경 시 유출 비밀번호 차단. 비밀번호 재설정 토큰은 1시간 만료.

SSO/SAML/OAuth, 2단계 인증, IP 제한은 MVP 범위 외 — 후속 단계에서 도입한다.

**이유**: SaaS 인증의 가장 단순하고 검증된 조합이다. SSO 도입은 엔터프라이즈 고객 진입 시점에 맞춰 한 번에 하면 된다.

### 5.12 P0 화면 목록 잠금

**결정**: MVP 핵심 흐름에 반드시 필요한 P0 화면 목록(로그인, 회원가입, 이메일 인증, 초대 수락, 조직 생성, 홈 대시보드 골격, 접근 제한, 계정 설정 기본형, 관리자 홈 골격)을 Phase A 문서 §6.1로 잠갔다.

**이유**: B단계 착수 시점의 화면 범위가 흔들리면 일정 추정이 의미를 잃는다.

### 5.13 Phase B 최소 데이터 엔티티

**결정**: B단계가 만들어야 할 데이터베이스 테이블을 다음으로 잠갔다.

* `users`(사용자), `sessions`(로그인 세션), `invitations`(조직 초대)
* `organizations`(조직), `teams`(팀), `memberships`(사용자 ↔ 조직 연결), `roles`(역할 정의)
* `activity_logs`(활동 로그) — 골격만(테이블과 작성 인터페이스만; 실제 광범위한 기록은 Phase G에서 추가)
* `organization_settings`(조직 기본 설정) — 핵심 컬럼만

문서, AI, 검색, 업무 맥락 테이블은 **B단계에서 만들지 않는다**.

**이유**: B단계 범위를 좁게 잡아야 일정 안에 인증·조직·권한 골격이 안정적으로 끝난다. 다음 단계에서 필요해질 테이블을 미리 만들어두면 바뀔 가능성이 높아 낭비가 된다.

## 6. 미루기로 한 항목 — Phase A §16 Deferred

다음은 MVP에 **들어가지 않는다**고 명시한 항목들이다. B–H 단계에서 슬그머니 들어오는 것을 막기 위해 별도 표로 박아 두었다.

| 항목 | 미룬 이유 |
| --- | --- |
| 다중 조직 소속 | 조직 전환 UI와 권한 복잡도 증가 |
| 다중 팀 소속 | 별도 조인 테이블과 권한 쿼리 재설계 필요 |
| Department 별도 개념 | 단일 팀이 MVP 수요를 모두 커버 |
| 외부 링크 공유 | 통제된 접근 모델 밖 |
| Manager의 공유 승인 워크플로 | Manager가 팀 범위는 직접 공유 |
| Admin 본문 열람 / 컴플라이언스 접근 | 신뢰 민감 — 별도 설계와 법무 검토 필요 |
| 팀 협업 To-do | 할당/알림/권한 다툼 등 부수 기능 따라옴 |
| 첨부 파일 | 저장, 안티바이러스, 미리보기 파이프라인 — 별도 단계 |
| 문서 버전 복원 UI | 버전 저장은 MVP, 복원 UI는 P1 |
| SSO / SAML / OAuth | 엔터프라이즈 영업 시점에 한 번에 |
| 2FA / TOTP | 출시 후 우선순위 |
| 실시간 협업 편집 | MVP 범위 외 |
| 자동 AI Agent | MVP 범위 외 |
| 모바일 / 데스크톱 / 온프레미스 | MVP 범위 외 |
| AI 본문 기본 저장 | 보안 — opt-in만 허용 |

## 7. Phase B 진입 기준 — Phase A §17

다음 조건이 **모두** 만족되어야 B단계를 시작할 수 있다고 잠갔다.

1. §15 결정이 본 문서에 잠겨 있다.
2. §16 Deferred 항목이 기록되어 있고 B단계 범위에 들어가지 않는다.
3. P0 화면 목록(§6.1)이 잠겨 있다.
4. B단계 최소 엔티티가 데이터베이스 설계 문서에 반영되어 있다.
5. 인증 방식, 기본 공유 범위, 삭제 정책, AI 로그 보존 정책이 보안·AI 정책 문서에 반영되어 있다.
6. Codex가 §15 결정을 권한 정책·DB 설계 문서에 대해 검증 완료했다.

이 6개는 Phase A 작업으로 모두 충족했다.

## 8. 정합성 패치 — 결정을 다른 문서에 반영

§15 결정을 잠그는 것만으로는 부족하다. 보안 정책, DB 설계, AI 정책, API 명세, 그리고 다른 단계 계획서가 모두 같은 말을 해야 한다. Phase A 후반에 다음 8개 문서를 정합화했다.

### 8.1 보안 정책 (`docs/security/notive-permission-policy-v1.0.md`)

* §6.3 문서 조회 권한에서 "Admin 전체 열람 옵션이 켜져 있으면 Admin이 본문 조회 가능"이라는 옵션형 문구를 **삭제**했다. Admin은 메타데이터만 본다는 점을 명문화했다.
* §8 To-do 권한 절을 개인 전용으로 좁혔다. 상태값을 `todo` / `in_progress` / `done`으로 명시했다.
* §9.5 "AI 문제 신고 페이로드 접근 권한" 절을 신설했다. 제품 화면 어디에서도 본문이 노출되지 않으며, 운영 비상 절차(break-glass)로만 접근 가능하고 모든 접근은 별도 감사 로그로 기록한다.
* §11.1 관리자 권한 표를 갱신해 Manager의 사용자 초대와 템플릿 관리를 **불가**로 잠갔다. (이 두 결정은 Phase B 작업 시점에 Codex가 추가로 잠근 결정이지만, Phase A의 Manager 범위 결정과 자연스럽게 이어지는 부분이라 보고서에서 함께 언급한다.)
* §15 오류 코드 정책 — 권한 없는 리소스의 기본 응답을 `NOT_FOUND`로 잠갔다(`FORBIDDEN`은 인증되었지만 기능 권한이 없는 경우만).
* §17 미결정 표를 §17 결정 표로 옮기고, 잔여 미결정 항목만 §17.1에 분리했다.

### 8.2 데이터베이스 설계 (`docs/database/notive-database-design-v1.0.md`)

* §6.3 `memberships` 테이블에 "MVP는 단일 팀 소속" 잠금을 명문화했다. 다중 팀은 후속 단계의 별도 조인 테이블(`membership_teams`)로 도입한다고 메모로 남겼다.
* §10.3 `todos` 테이블을 개인 전용으로 재설계했다. `assignee_user_id` / `team_id` / `visibility` / `Cancelled` 상태를 모두 제거하고 `owner_user_id` 단일 + `todo`/`in_progress`/`done` 3상태로 단순화했다. 향후 협업 To-do 도입 시 추가될 컬럼 목록을 메모로 남겼다.
* §9.1 `ai_requests`에서 `request_text` 컬럼을 **제거**했다. 사용자 요청 원문은 영구 저장하지 않는다.
* §9.2 `ai_results`에서 `title` / `content` / 자유 형식 `error_message`를 **제거**했다. 결과 본문은 단기 스토리지에서만 살고, 자유 메시지는 코드값(`error_code`)만 저장한다.
* §9.4 `ai_request_payloads` 테이블을 **신설**했다. 사용자가 명시적으로 신고한 경우에만 30일간 본문을 보관하는 opt-in 테이블이다.
* §15.3 AI 본문 보존 정책을 명시했다(메타데이터 90일, 페이로드 30일, 미리보기 24시간 단기 스토리지, Admin 본문 미노출).
* §16 미결정 표를 결정/잔여 미결정 두 표로 분리했다.

### 8.3 AI 생성 정책 (`docs/ai/notive-ai-generation-policy-v1.0.md`)

* §12 로그 및 보존 정책을 5개 소절로 재작성해 잠갔다. 기본 메타데이터 90일, 미저장 항목(요청 본문, 응답 본문, 참고 자료 본문, 자유 형식 오류 메시지), opt-in 신고 페이로드 30일, 관리자 노출 제한(메타만), 정리 작업 운영 원칙.
* §15 미결정 표에서 결정된 항목을 §15 결정 표로 옮기고, 진짜 미결정만 §15.1에 남겼다.

### 8.4 API 명세 (`docs/api/notive-api-spec-v1.0.md`)

* §3.5 공통 오류 코드 표 아래에 권한 거부 응답 규칙(NOT_FOUND 기본, FORBIDDEN은 인증된 사용자의 기능 권한 부족, 마지막 Admin 보호는 `FORBIDDEN(reason_code=last_admin_protection)`)을 명시했다.
* §9 AI 문서 생성 API 머리말에 "본문은 영구 테이블에 저장되지 않으며 단기 스토리지에 보관, 24시간/폐기 시 즉시 삭제, 신고 시에만 30일 보관"을 명시했다.
* §10.4–10.6 To-do API에서 `assigneeUserId` / `teamId` / `visibility`를 제거하고 상태값을 소문자(`todo` / `in_progress` / `done`)로 통일했다.
* §13 권한 요약 표를 갱신해 Manager의 초대/템플릿 관리 불가, To-do 본인 전용 등을 반영했다.

### 8.5 Phase C 문서 관리 계획 (`docs/implementation/notive-implementation-plan-c-document-management-v1.0.md`)

* §8.3 공유 범위 표에서 "Department" 행을 삭제했다.
* §8.4 권한 예외 원칙에서 "Admin 전체 문서 접근 허용 여부는 미결정"이라는 표현을 "Admin은 묵시적 본문 접근 미허용, 메타데이터만 조회"로 정정했다.

### 8.6 Phase D AI 문서 생성 계획 (`docs/implementation/notive-implementation-plan-d-ai-document-generation-v1.0.md`)

## 11. 남은 리스크와 향후 결정 필요 항목

Phase A에서 잠근 결정에는 다음과 같은 운영 리스크와 잔여 결정이 남아 있다. B단계 이후에 다시 마주칠 가능성이 있다.

### 11.1 운영 리스크

* **단기 스토리지 운영** — AI 미리보기 본문을 24시간만 단기 스토리지에 두는 정책은 인프라(예: Redis) 수준의 TTL 설정과 정리 작업이 정확히 동작해야 의미가 있다. B단계 인프라 구현 단계에서 검증이 필요하다.
* **AI 품질 회귀 분석 데이터 부족 가능성** — 본문이 영구 저장되지 않으므로 사용자가 신고하지 않은 AI 품질 문제는 추적이 어렵다. 출시 후 신고 데이터가 충분히 모이는지 모니터링이 필요하다.
* **Last-Admin 보호의 코드 우회 가능성** — DB 트리거와 애플리케이션 양쪽에서 막아야 안전하다. B단계 마이그레이션 단계에서 raw SQL로 트리거를 추가하기로 했다.
* **다중 팀 소속 도입 비용** — 후속 단계에서 도입할 경우 스키마 변경(별도 조인 테이블)뿐 아니라 권한 쿼리 전반의 재작성이 필요하다. 비용이 결코 작지 않으므로 도입 결정을 가볍게 내리지 않는다.

### 11.2 잔여 미결정 (Phase A에서 결정하지 않음)

* **ID 타입** — UUID vs bigint. DB 마이그레이션 단계에서 결정.
* **문서 본문 저장 형식** — Markdown / HTML / JSON document. C단계 시작 전 결정.
* **검색 인덱스 방식** — DB 기반 vs 별도 검색 엔진. F단계에서 결정.
* **AI 조직별 문체 설정 도입 시점**.
* **AI 재생성 횟수 제한 정책**.
* **단기 스토리지 매니지드 서비스 선정** — Redis / ElastiCache / Upstash / self-hosted 등.

이들은 Phase A에서 정하지 않아도 B단계 착수에 지장이 없으므로 잔여 미결정으로 남겼다. 각자 해당 단계가 시작될 때 다시 다룬다.

## 12. 비개발자 요약 — 한 단락

Phase A는 "코드 한 줄도 안 짜고 미리 결정해두지 않으면 다음 단계에서 비싸게 후회할 12가지를 못박은 단계"다. 가입 방식, 관리자 권한 범위, 데이터 보관 기간, AI 본문 저장 정책 같은 결정을 잠그고, 그 결정들이 보안 정책 / DB / AI / API 문서 모두에서 같은 말을 하게 정리했다. 이 작업이 끝났기 때문에 다음 단계(B단계 — 실제 서비스 기반 구축)가 안전하게 시작될 수 있다.
