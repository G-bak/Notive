# 06. API 명세서 작업 보고서

## 대상 문서

`docs/api/notive-api-spec-v1.0.md`

서버와 클라이언트(웹 앱)가 주고받는 API의 형태를 정의한 문서다. 어떤 URL로 어떤 데이터를 보내면 어떤 응답이 오는지의 계약이다.

## 변경한 내용

### 1. AI 문서 생성 API에 본문 저장 정책 명시

§9 첫머리에 다음을 추가했다.

> 본 절의 요청/응답 본문(`requestText`, `title`, `content`)은 영구 테이블에 저장되지 않는다. 미리보기/편집 단계의 본문은 세션 바운드 단기 스토리지에만 보관되며, 사용자가 §9.4(저장 API)로 저장해야 `documents`에 영속화된다. 미저장 24시간 또는 폐기 시 단기 스토리지에서 즉시 삭제한다.

§9.1 AI 생성 요청 API의 `requestText` 필드 아래에 다음 주석도 추가했다.

> `requestText`는 `ai_requests`에 영구 저장되지 않는다. 사용자가 "문제 신고" opt-in 흐름을 사용한 경우에만 `ai_request_payloads`에 30일 한도로 저장된다.

API 자체의 요청/응답 형태는 그대로 유지했다(클라이언트 입장에서는 동일). 다만 서버 안에서 무엇이 어떻게 저장되는지를 문서에 박았다.

### 2. To-do API를 개인 전용으로 정정

#### §10.4 To-do 목록 조회

- 응답이 항상 요청 사용자가 만든 항목만 반환한다는 점을 명시.
- 쿼리 파라미터에서 `assigneeId`(담당자), `teamId`(팀) 제거.
- 상태 파라미터 값을 소문자(`todo`, `in_progress`, `done`)로 통일.

#### §10.5 To-do 생성

- 요청 본문에서 `assigneeUserId`, `visibility` 제거.
- "생성자가 곧 소유자다(개인 To-do)"라는 권한 메모 추가.

#### §10.6 To-do 상태 변경

- 권한을 "생성자 본인만"으로 명시.
- 상태값을 소문자(예: `done`)로 변경.

## 변경 이유

- **AI 본문 저장 정책 명시**: API 명세서를 보고 구현하는 개발자가 "당연히 ai_requests에 본문도 저장하겠지"라고 추정하지 않도록 명세에 박았다. DB 설계와 API 명세가 같은 말을 해야 한다.
- **To-do 단순화**: Phase A §15에서 "개인 To-do만"이라고 결정했으므로 API도 개인 작업만 처리하는 형태로 좁혔다. 담당자 할당이나 팀 공개 같은 매개변수가 API에 남아 있으면 클라이언트가 그걸 보내고 서버는 무시하는 어색한 상황이 생긴다.
- **상태값 소문자 통일**: DB와 보안 정책에서 모두 소문자(`todo`/`in_progress`/`done`)를 쓰는데 API만 카멜케이스(`Done`)면 변환 코드가 곳곳에 생긴다. 한 가지 표기로 통일.

## B단계 착수에 미치는 영향

- B단계는 인증/조직/팀까지 만들고 To-do는 P1이라 직접 구현하지 않지만, **API 명세가 일관**되어 있어 후속 단계에서 To-do 구현 시 혼선이 없다.
- AI 본문 저장 정책이 API 명세에 박혀 있어 D단계 AI 기능 구현 시 "본문은 DB에 안 적는다"는 규칙을 자연스럽게 따르게 된다.
- B단계의 인증/조직 API는 이번에 손대지 않아 그대로다.

## 남은 주의사항

- API 명세서 전체가 아직 한국어다. 향후 영문화가 필요하면 별도 작업으로 진행한다(이번 Phase A 작업 범위에는 미포함).
- 권한 오류 코드(NOT_FOUND vs FORBIDDEN)에 대한 결정이 보안 정책 §17.1에 미결정으로 남아 있다. API 응답 코드와도 직결되므로 이 결정이 내려지면 API 명세도 다시 손봐야 한다.
- AI API의 응답 본문에 `title`/`content`가 그대로 반환되는데, 이 값이 단기 스토리지에서 오는 것임을 클라이언트 측 구현 시에도 인지해야 한다(클라이언트가 단기 스토리지의 만료를 인지하고 사용자에게 적절한 안내를 띄울 필요가 있다).
