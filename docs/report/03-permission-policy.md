# 03. 권한 정책 명세서 작업 보고서

## 대상 문서

`docs/security/notive-permission-policy-v1.0.md`

"누가 어떤 데이터를 보고 고치고 공유할 수 있는가"를 정의하는 문서다. 보안과 직결된다. 개발 시 모든 API와 화면이 이 문서의 규칙을 따라야 한다.

## 변경한 내용

### 1. Admin이 모든 문서 본문을 보는 권한 제거

기존: 조직 정책에 "Admin 전체 열람"이 켜져 있으면 Admin이 모든 문서 본문을 볼 수 있다.

변경: 그 옵션 자체를 없앴다. **Admin도 일반 사용자와 같은 권한 규칙을 따른다**. 단, 관리 콘솔에서 메타데이터(제목, 소유자, 공유 범위, 크기, 수정 시각)는 볼 수 있다. 본문은 못 본다.

### 2. To-do 권한을 개인 전용으로 축소

기존: To-do에 팀 공개 범위가 있고, Manager나 Admin이 팀 범위 To-do를 볼 수 있었다.

변경:
- 사용자는 **본인이 만든 To-do만 본다**.
- Manager/Admin도 다른 사용자의 To-do를 보지 않는다.
- 상태값은 `todo`, `in_progress`, `done` 세 가지만 쓴다.

### 3. AI 문제 신고 페이로드 접근 권한을 §9.5에 신설

사용자가 AI 결과에 문제가 있어 "문제 신고"를 누르면 그 요청의 본문이 30일 보관된다. 이 본문은:

- **제품 화면 어디에서도 보이지 않는다**. 사용자 본인도, Admin도 다시 못 본다.
- 진짜 인시던트 조사가 필요한 경우에만 운영자가 "비상 절차(break-glass)"로 접근할 수 있다.
- 이때 누가, 왜, 언제, 어떤 요청을 봤는지 별도 감사 로그에 남긴다.
- 30일이 지나면 자동으로 영구 삭제된다.

### 4. §17 미결정 표 정리

Phase A에서 결정된 항목(Admin 본문 열람, To-do 범위, 업무일지 공개 범위, AI 로그 본문 보관)을 미결정 표에서 빼고 결정 표로 옮겼다. 진짜 미결정인 항목만 §17.1에 남겼다.

## 변경 이유

- **Admin 묵시적 본문 열람 제거**: Notive는 회사 내부 문서를 다룬다. Admin이 "관리자니까 다 볼 수 있다"는 가정은 사용자 신뢰를 깬다. 일반 권한 규칙을 따르게 하면 문서 작성자가 "내 글이 누구에게 노출되는가"를 정확히 예측할 수 있다.
- **To-do 개인 전용**: MVP에서 팀 To-do까지 만들면 할당, 변경 알림, 권한 다툼 같은 부수 기능이 필요해진다. 일정과 가치가 안 맞아 후속으로 미뤘다.
- **페이로드 접근 권한 신설**: 새로 만든 `ai_request_payloads` 테이블에 회사 내부 정보가 그대로 들어가므로 누가 언제 볼 수 있는지를 보안 정책에 명시적으로 박아야 한다. 그렇지 않으면 운영 중 무심코 노출된다.

## B단계 착수에 미치는 영향

- B단계는 권한 모듈을 만들 때 이 문서의 §6.3, §8, §9.5를 그대로 따른다.
- "Admin이라도 본문은 안 보인다"는 규칙이 명확하므로 권한 검사 코드가 단순해진다(`if user.is_admin: return True` 같은 우회로가 없다).
- To-do API는 본인 데이터만 반환하므로 쿼리에 항상 `owner_user_id = current_user` 조건을 붙이면 된다.

## 남은 주의사항

- §17.1 잔여 미결정(Manager 사용자 초대, Manager 템플릿 관리, 권한 오류 코드)은 B/C단계 작업 중 다시 결정해야 한다.
- 페이로드 비상 접근 절차는 보안 정책에는 적었지만, **누가 운영자 권한을 가질지, 어떻게 신청할지**의 운영 절차는 별도 운영 가이드에서 마무리해야 한다.
- 향후 컴플라이언스(예: 법적 자료 제출 요청) 대응 시 Admin 본문 열람을 다시 검토해야 할 수 있다. 그때는 별도 "컴플라이언스 액세스" 기능으로 설계해야 하며, 묵시적 권한 부활은 안 된다.
