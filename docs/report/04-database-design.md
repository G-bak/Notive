# 04. DB 설계서 작업 보고서

## 대상 문서

`docs/database/notive-database-design-v1.0.md`

데이터베이스에 어떤 표(table)를 만들고 각 표에 어떤 칸(field)을 둘지 정의하는 설계 문서다. 실제 DB 마이그레이션 SQL이 아니라, "어떤 데이터를 어떻게 보관할지"의 계약이다.

## 변경한 내용

가장 변경이 많은 문서다. 핵심은 **AI 본문을 영구 저장 테이블에서 제거**한 것과 **To-do를 개인 전용으로 단순화**한 것이다.

### 1. 멤버십(memberships) 테이블에 단일 팀 잠금 명시

`memberships` 테이블에 사용자가 속한 팀 정보를 저장하는 컬럼이 하나 있는데(`team_id`), 이 컬럼이 **MVP에서는 단 하나의 팀만 가리킨다**는 것을 문서에 명시했다. 한 사용자가 여러 팀에 속하려면 별도 조인 테이블(`membership_teams`)이 필요한데, 그건 후속 단계로 미룬다.

### 2. To-do 테이블을 개인 전용으로 다시 설계

기존 `todos` 테이블에는 다음 칸들이 있었다.

- `assignee_user_id`(담당자), `team_id`(팀), `visibility`(공개 범위), `status`에 `Cancelled` 포함, `NotStarted`/`InProgress`/`Done` 같은 카멜케이스 상태값.

이번에 다음과 같이 바꿨다.

- 담당자/팀/공개 범위 칸을 **삭제**(MVP는 개인 To-do만 지원).
- `owner_user_id` 한 칸이 생성자이자 소유자 역할.
- 상태값은 소문자 `todo`, `in_progress`, `done` 세 가지만.
- "미래에 팀 협업 To-do를 만들 때 추가될 칸 목록"을 표 아래에 메모로 남겼다(아예 잊지 않도록).

### 3. AI 요청 테이블에서 사용자 요청 원문 제거

기존 `ai_requests` 테이블에는 `request_text`라는 칸이 있어서 사용자가 AI에게 보낸 요청 문장을 그대로 저장하고 있었다.

이번에 그 칸을 **DB에서 제거**했다. 영구 저장 안 함.

### 4. AI 결과 테이블에서 결과 본문/제목 제거

기존 `ai_results` 테이블에는 `title`, `content`, `error_message` 칸이 있었다.

이번에 셋 다 **DB에서 제거**했다. 결과 본문은 사용자가 미리보기/편집하는 동안 단기 스토리지(예: Redis)에 잠깐 보관하다가, 사용자가 "저장" 버튼을 누르면 `documents` 테이블로 옮겨진다. 미저장으로 24시간 지나거나 사용자가 폐기하면 즉시 삭제된다.

### 5. 새 테이블 `ai_request_payloads` 신설

사용자가 AI 결과에 문제가 있어 "문제 신고"를 누른 경우에만 그 요청의 본문(prompt + response + 참고 자료 스냅샷)을 임시로 저장하는 테이블이다.

- 30일 후 자동 삭제(`retain_until` 칸).
- 신고 사용자 ID와 신고 사유 코드를 함께 저장.
- 보안 정책 §9.5와 직결됨: 제품 화면 어디에서도 본문이 노출되지 않는다.

### 6. §15 보존 정책과 §16 미결정 표 정리

§15에 AI 본문 보존 규칙(메타데이터 90일, 페이로드 30일)을 명시. §16은 결정된 항목과 잔여 미결정 항목을 두 표로 분리했다.

## 변경 이유

- **AI 본문 저장을 막은 이유**: AI 요청에는 회사의 미공개 정보(고객명, 금액, 인사 정보 등)가 그대로 들어갈 수 있다. 그게 영구 테이블에 쌓이면 운영자 실수로 보거나, 향후 데이터 유출 사고 시 피해가 커진다. **"보관하지 않는 것이 가장 안전한 보관"**이라는 원칙으로 기본을 미저장으로 잡았다.
- **그래도 신고 본문은 잠깐 저장하는 이유**: AI가 부적절한 결과를 만들었을 때 무엇이 잘못됐는지 운영팀이 봐야 고칠 수 있다. 사용자가 명시적으로 동의한 단일 요청에 한해 30일만 보관한다.
- **To-do 단순화**: Phase A §15의 "개인 To-do만"이라는 결정과 정합을 맞춘 것이다. DB가 협업 기능을 가정한 채로 남아 있으면 코드에서 그 가정을 따라가게 된다.
- **단일 팀 명시**: 다중 팀 소속을 지원한다고 적혀 있으면서 단일 팀 컬럼만 두는 건 모순이다. 결정을 단일 팀으로 강하했으므로 문서도 그에 맞췄다.

## B단계 착수에 미치는 영향

- B단계의 **마이그레이션 SQL은 이 새 스키마를 기준으로** 작성한다. 특히 `ai_request_payloads`는 신규 테이블이므로 마이그레이션에 포함시켜야 한다.
- B단계에서 만들 테이블은 §3 "테이블 그룹"에서 인증/조직 그룹만 해당한다. AI/문서/검색 그룹은 후속 단계에서 만든다.
- AI 결과 본문이 영구 DB에 없으므로, 미리보기를 위한 단기 스토리지(Redis 등) 인프라 결정이 B단계 인프라 작업에 포함되어야 한다.
- `todos` 스키마가 단순해져서 B단계의 권한 쿼리·인덱스 설계가 한결 수월해진다.

## 남은 주의사항

- 단기 스토리지의 구체 인프라(Redis vs 다른 캐시)와 운영 방식(암호화, 백업 정책, 만료 처리 작업)이 아직 결정되지 않았다. B단계 인프라 결정에서 마무리해야 한다.
- AI 본문이 영구 저장되지 않으면 **AI 품질 문제 분석이 어려워진다**. opt-in 신고가 충분한 양의 데이터를 모을 수 있는지 운영 시작 후 확인이 필요하다(Phase H 출시 전 점검 항목).
- `memberships.team_id` 단일성은 문서로는 잠갔지만, **DB 제약(unique, check constraint 등)으로 강제하는 부분은 마이그레이션 단계에서 다시 점검**해야 한다. 문서 수준의 약속은 코드 단계에서 깨질 수 있다.
- §16.1 잔여 미결정(ID 타입 UUID/bigint 선택, 문서 본문 저장 형식, 검색 인덱스 방식 등)은 B/F단계에서 다시 결정해야 한다.
