# 권한 정책 명세서 v1.0

# Notive Permission Policy

---

# 1. 문서 목적

본 문서는 Notive MVP의 권한 정책을 정의한다.

권한 정책은 문서 조회/수정/공유, AI 참고 자료 사용, 검색 결과 노출, 업무 기록 접근, 관리자 기능 접근에 공통으로 적용된다. 이 문서는 구현 시 Permission Module, API 권한 검사, 화면 접근 제어, QA 테스트의 기준으로 사용한다.

---

# 2. 권한 설계 원칙

## 2.1 서버 기준 권한 판단

권한은 화면에서만 제어하지 않는다.

모든 보호 API와 데이터 조회 단계에서 서버가 최종 권한을 판단한다.

---

## 2.2 조직 경계 우선

모든 리소스는 조직 경계를 가진다.

사용자는 자신이 Active 멤버로 속한 조직의 데이터만 접근할 수 있다.

---

## 2.3 최소 권한

사용자에게 필요한 최소 권한만 부여한다.

권한이 애매한 경우 허용보다 차단을 기본값으로 한다.

---

## 2.4 AI와 검색에도 동일한 권한 적용

문서 상세에서 볼 수 없는 자료는 검색 결과에도 나오면 안 되며, AI 생성 참고 자료로도 사용할 수 없다.

---

## 2.5 민감 정보 존재 여부 비노출

권한 없는 문서는 존재 여부를 암시하지 않는다.

검색 결과 없음과 권한 없음은 사용자에게 과도한 정보를 주지 않도록 처리한다.

---

# 3. 기본 역할

## 3.1 역할 목록

| 역할 | 설명 |
| --- | --- |
| Viewer | 허용된 문서를 조회하는 사용자 |
| Editor | 문서 작성, 업무 기록, AI 생성이 가능한 일반 사용자 |
| Manager | 팀 범위 문서와 사용자 일부를 관리하는 사용자 |
| Admin | 조직 전체 설정과 사용자/권한을 관리하는 사용자 |

---

## 3.2 역할 계층

역할은 기본적으로 다음 계층을 따른다.

```text
Viewer < Editor < Manager < Admin
```

단, 모든 권한이 자동 상속되는 것은 아니다. 문서 접근은 역할뿐 아니라 공유 범위, 소유자, 팀, 개별 공유 설정을 함께 판단한다.

---

# 4. 공통 접근 조건

모든 보호 리소스 접근 전 다음 조건을 확인한다.

1. 사용자가 로그인 상태인가?
2. 사용자의 계정 상태가 Active인가?
3. 사용자가 해당 조직의 Active 멤버인가?
4. 요청 리소스가 해당 조직에 속하는가?
5. 사용자의 역할과 리소스 권한이 요청 작업을 허용하는가?

하나라도 만족하지 않으면 접근을 거부한다.

| To-do | 불가 | 가능 | 가능 | 가능 |

| 설정 | 가능 | 가능 | 가능 | 가능 |

메뉴가 숨겨져 있어도 직접 URL 접근은 서버에서 다시 차단한다.

# 6. 문서 권한 정책

## 6.1 문서 공유 범위

| 공유 범위 | 설명 |
| --- | --- |
| Private | 소유자 또는 작성자 중심 접근 |
| Team | 지정 팀 사용자 접근 |
| Organization | 조직 전체 사용자 접근 |
| SpecificUsers | 지정 사용자 접근 |

MVP에서는 Department를 별도 개념으로 두지 않고 Team으로 처리한다.

---

## 6.2 문서 권한 수준

| 권한 | 설명 |
| --- | --- |
| View | 문서 조회 |
| Edit | 문서 수정 |
| Manage | 문서 공유, 삭제, 복원 등 관리 |

---

## 6.3 문서 조회 권한

사용자는 다음 중 하나를 만족하면 문서를 조회할 수 있다.

* 문서 소유자다.
* 문서 작성자다.
* 문서가 Organization 공개이고 같은 조직 멤버다.
* 문서가 Team 공개이고 사용자가 해당 팀에 속한다.
* document_shares에 사용자 또는 팀 대상으로 View 이상 권한이 있다.
* Admin 전체 열람 정책이 켜져 있고 사용자가 Admin이다.

---

## 6.4 문서 작성 권한

| 역할 | 작성 가능 여부 |
| --- | --- |
| Viewer | 불가 |
| Editor | 가능 |
| Manager | 가능 |
| Admin | 가능 |

---

## 6.5 문서 수정 권한

사용자는 다음 중 하나를 만족하면 문서를 수정할 수 있다.

* 문서 소유자이며 Editor 이상이다.
* document_shares에 Edit 이상 권한이 있다.
* Manager이고 문서가 자신의 팀 범위에 속하며 팀 관리 정책이 허용한다.
* Admin이고 조직 정책상 문서 수정이 허용된다.

---

## 6.6 문서 관리 권한

공유 설정, 삭제, 복원, 소유권 변경은 Manage 권한으로 본다.

사용자는 다음 중 하나를 만족하면 문서를 관리할 수 있다.

* 문서 소유자다.
* document_shares에 Manage 권한이 있다.
* Manager이고 팀 문서 관리가 허용되어 있다.
* Admin이고 조직 정책상 문서 관리가 허용된다.

---

## 6.7 문서 권한 예외

* Deleted 상태 문서는 일반 사용자 목록에 표시하지 않는다.
* Archived 상태 문서는 권한은 유지하되 기본 목록에서 제외할 수 있다.
* 다른 조직의 문서는 어떤 역할도 접근할 수 없다.
* 직접 URL 접근 시에도 문서 권한을 다시 확인한다.

---

# 7. 업무 다이어리 권한 정책

## 7.1 업무 기록 공개 범위

| 공개 범위 | 조회 가능 대상 |
| --- | --- |
| Private | 작성자 |
| Team | 작성자와 같은 팀 사용자 |
| Organization | 조직 전체 사용자 |

MVP에서는 Private과 Team을 우선 제공한다.

---

## 7.2 업무 기록 작성 권한

| 역할 | 작성 가능 여부 |
| --- | --- |
| Viewer | 불가 |
| Editor | 가능 |
| Manager | 가능 |
| Admin | 가능 |

---

## 7.3 업무 기록 수정/삭제 권한

* 작성자는 자신의 업무 기록을 수정/삭제할 수 있다.
* Admin의 전체 수정/삭제 권한은 운영 정책에 따라 제한할 수 있다.
* 다른 사용자의 Private 기록은 Manager와 Admin도 기본적으로 조회하지 않는 방향을 권장한다.

---

# 8. To-do 권한 정책

## 8.1 To-do 조회 권한

사용자는 다음 중 하나를 만족하면 To-do를 조회할 수 있다.

* To-do 생성자다.
* To-do 담당자다.
* To-do가 Team 공개이고 같은 팀 사용자다.
* Manager이고 팀 범위 To-do 조회가 허용된다.
* Admin이고 조직 정책상 조회가 허용된다.

---

## 8.2 To-do 작성 권한

| 역할 | 작성 가능 여부 |
| --- | --- |
| Viewer | 불가 |
| Editor | 가능 |
| Manager | 가능 |
| Admin | 가능 |

---

## 8.3 To-do 수정 권한

* 생성자는 수정 가능하다.
* 담당자는 상태 변경 가능하다.
* Manager는 팀 범위 To-do를 수정할 수 있다.
* Admin은 조직 정책상 수정할 수 있다.

# 9. AI 생성 권한 정책

## 9.1 AI 문서 생성 권한

| 역할 | AI 생성 가능 여부 |
| --- | --- |
| Viewer | 불가 |
| Editor | 가능 |
| Manager | 가능 |
| Admin | 가능 |

---

## 9.2 AI 참고 자료 권한

AI 생성에 사용할 수 있는 참고 자료는 사용자가 접근 가능한 자료로 제한한다.

### 대상별 기준

| 대상 | 권한 기준 |
| --- | --- |
| Document | 문서 View 이상 권한 필요 |
| Template | 템플릿 적용 범위에 포함되어야 함 |
| DiaryEntry | 업무 기록 조회 권한 필요 |
| Todo | To-do 조회 권한 필요 |

---

## 9.3 AI 요청 시 권한 검사

AI 생성 요청 시 다음 검사를 수행한다.

1. 요청 사용자가 Editor 이상인지 확인한다.
2. 선택된 템플릿을 사용할 수 있는지 확인한다.
3. 모든 참고 자료에 대해 조회 권한을 확인한다.
4. 참고 자료가 현재 조직에 속하는지 확인한다.
5. 삭제 또는 비활성 상태 자료를 제외한다.

---

## 9.4 AI 결과 접근 권한

* AI 생성 결과는 요청 사용자만 기본 접근 가능하다.
* AI 결과가 문서로 저장되면 문서 권한 정책을 따른다.
* Admin이 AI 요청 기록을 볼 수 있는 범위는 로그 정책에 따른다.

---

# 10. 검색 권한 정책

## 10.1 검색 가능 대상

검색 결과는 사용자가 조회 가능한 리소스만 포함한다.

| 대상 | 검색 노출 기준 |
| --- | --- |
| Document | 문서 View 권한 필요 |
| DiaryEntry | 업무 기록 조회 권한 필요 |
| Todo | To-do 조회 권한 필요 |

---

## 10.2 검색 결과 권한 처리

* 권한 없는 문서는 검색 결과에 표시하지 않는다.
* 권한 없는 문서의 존재를 암시하지 않는다.
* 검색 결과에서 원문 열기 시 권한을 다시 확인한다.
* 검색 인덱스 조회에도 조직 ID 조건을 적용한다.

---

## 10.3 AI 요약 검색 권한

AI 요약 검색은 접근 가능한 출처만 사용한다.

출처가 없거나 권한 있는 근거가 부족하면 답변을 생성하지 않는다.

---

## 11.2 Admin 보호 정책

* 마지막 Admin은 비활성화할 수 없다.
* 마지막 Admin의 역할을 낮출 수 없다.
* Admin이 자기 자신의 권한을 낮추는 작업은 재확인 또는 차단한다.
* 역할 변경은 ActivityLog에 기록한다.

---

## 11.3 Manager 제한 정책

Manager는 기본적으로 팀 범위만 관리한다.

MVP에서는 Manager의 관리 기능을 보수적으로 제한하고, 필요한 경우 후속으로 확장한다.

# 15. 권한 오류 처리

## 15.1 오류 코드

| 상황 | 코드 |
| --- | --- |
| 로그인 필요 | UNAUTHORIZED |
| 조직 멤버 아님 | FORBIDDEN |
| 역할 부족 | FORBIDDEN |
| 리소스 권한 없음 | NOT_FOUND 또는 FORBIDDEN |
| 다른 조직 리소스 | NOT_FOUND 또는 FORBIDDEN |

---

## 15.2 NOT_FOUND와 FORBIDDEN 사용 기준

권한 없는 리소스의 존재를 숨겨야 하는 경우 `NOT_FOUND`를 사용할 수 있다.

예:

* 다른 조직 문서
* 권한 없는 Private 문서
* 검색 결과에서 권한 없는 문서

역할 부족처럼 사용자가 권한 부족을 이해해야 하는 경우 `FORBIDDEN`을 사용할 수 있다.

예:

* Viewer가 AI 생성 화면 접근
* Editor가 관리자 화면 접근

---

# 16. 권한 테스트 체크리스트

## 16.1 조직 경계

* 다른 조직 문서를 조회할 수 없는가
* 다른 조직 문서가 검색되지 않는가
* 다른 조직 문서를 AI 참고 자료로 사용할 수 없는가
* 다른 조직 사용자 목록을 조회할 수 없는가

---

## 16.2 문서 권한

* Private 문서는 소유자만 조회 가능한가
* Team 문서는 같은 팀 사용자만 조회 가능한가
* SpecificUsers 문서는 지정 사용자만 조회 가능한가
* 권한 없는 문서는 목록에 표시되지 않는가
* 직접 URL 접근도 차단되는가

---

## 16.3 AI 권한

* Viewer는 AI 생성이 불가한가
* 권한 없는 문서를 참고 자료로 선택할 수 없는가
* 생성 직전 권한이 변경된 자료는 제외되는가
* AI 결과는 요청 사용자에게만 보이는가

---

## 16.4 검색 권한

* 권한 없는 문서는 검색 결과에 나오지 않는가
* AI 요약에 권한 없는 문서가 사용되지 않는가
* 출처 문서 열기 시 권한을 재검사하는가

---

## 16.5 관리자 권한

* Editor는 관리자 화면에 접근할 수 없는가
* Manager는 조직 전체 설정을 수정할 수 없는가
* 마지막 Admin 제거가 차단되는가
* 역할 변경이 ActivityLog에 기록되는가

| 리소스 권한 오류 코드 | NOT_FOUND 중심, FORBIDDEN 중심 | 보안과 UX |
